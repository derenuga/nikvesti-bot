# Досьє v2 + сутнісний шар — план (бриф для нової сесії)

**Статус:** план. Обрано канал виконання прогону — **через Claude Code під
Max** (безкоштовно з підписки, не API; див. §3.3.1). Олег апрувнув спершу
ТЕСТ на малій вибірці (~100 статей), повний прогін архіву — після перевірки
якості (критерії §3.6). Написано 04.07.2026, оновлено 10.07.2026. Самодостатній
бриф: нова сесія має змогти реалізувати без контексту старої. Промт для старту
нової сесії — наприкінці файлу (§5).

**Передумови (вже зроблено, хвиля A):** «лисяча нора» наповнена — 286 301
стаття 2009–2026 у Postgres бота (`articles`, роздільні text_ua/text_ru,
зважений FTS: заголовок A > теги B > текст C; `tags`/`article_tags`;
інкрементальний sync щогодини о :50). Див. `ARCHIVE_MIRROR_MODULE.md`.

## 1. Проблема-тригер

`/dossier Виталий Воронов` → «24 матеріали». Реально в норі про Воронова
**647** (`fts @@ 'Воронов:*'`). Дві незалежні вади:

1. **Досьє семплить.** Дизайн v1: ≤3 статті/рік, ≤22 всього, по 700 симв.
   ліду, один виклик Sonnet, вихід ≤4096 симв. (ліміт Telegram). Вибірка по
   ts_rank — але для запиту-імені ранг майже плоский (скандал і прохідна
   згадка ранжуються однаково) → вибірка фактично випадкова, показано 3.7%
   знайденого. Для «пірнути в минуле і дістати артефакти» — непридатно.
2. **Пошук по словах не знає сутностей.** «Воронов = депутат = міськрада»
   для FTS не існує. Згадка «Виталий» без прізвища в середині тексту не
   знаходиться. Відмінки/ру-уа форми частково закриті префіксами, але
   однофамільці й ролі — ні.

## 2. Досьє v2 — три яруси

### 2.1 Швидкі виправлення (кодяться без сутнісного шару, ~вечір)

- **Повний індекс-файл — головний хід.** Наратив і покриття — різні продукти.
  До повідомлення-таймлайну прикріплювати **Telegram-документ** (txt або html):
  УСІ знайдені матеріали хронологічно — «рік → дата — заголовок — URL».
  0 токенів (чиста вибірка з нори), 100% покриття, ліміт 4096 більше не
  диктує глибину. Журналіст «пірнає» по файлу, читає таймлайн у чаті.
- Чесний рядок: «знайдено 647, у таймлайні — ключові» (зараз число ковтається).
- **Пріоритет "стаття ПРО людину":** для наративної вибірки спершу статті, де
  запит влучає в ЗАГОЛОВОК (вага A вже в fts; практично — окремий запит по
  `to_tsvector('simple', title_ua||title_ru) @@ query` або `ts_rank` з вагами
  `'{0.1,0.2,0.4,1.0}'`), потім добивати body-hits. Буст `own_material=1`
  (буст, НЕ фільтр — колонка існує лише з ~2024, див. ARCHIVE_MIRROR_MODULE).
- Підняти стелю вибірки для наративу (напр. 5/рік, 40 всього) — файл покриває решту.

### 2.2 Глибоке досьє: map-reduce на вимогу (`/dossier_deep <тема>`)

Фонова задача (патерн — бекфіл: asyncio.create_task, прогрес редагуванням
повідомлення, можливість зупинки):

- **Map:** усі знайдені статті (напр. 647) хронологічно, пачками по ~30
  повних текстів → Haiku 4.5, structured output: на кожну статтю
  `{id, date, facts: "1-2 речення фактів про тему", significance: 1-5}`.
  Правило: тільки те, що текстуально є в статті.
- **Reduce:** усі мікрофакти (~647 × ~40 ткн ≈ 26к ткн) → Sonnet: розгорнутий
  наратив по періодах, «ключові артефакти», відкриті питання. Якщо фактів
  забагато — ієрархічний reduce: по періодах (2009–2014, 2015–2019, …),
  потім фінальне злиття.
- **Вихід:** HTML/MD-файл документом (повне досьє з лінками на кожен факт) +
  стисле резюме в чат. Правило чесності беків діє: факт без лінка не виходить.
- **Вартість:** ~647 статей × ~700 ткн ≈ 450к вх Haiku ($0.5–0.8) + reduce
  Sonnet ($0.1–0.2) ≈ **$1 і кілька хвилин** на глибоке досьє. Прийнятно для
  свідомого інструмента. Map-результати кешувати в норі
  (`dossier_facts(article_id, topic_hash, facts, significance)`) — повторний
  прогін тієї ж теми майже безкоштовний.
- **Map-фазу гнати через Batch API (−50%)** → ~$0.5/досьє. Батч повертається
  за хвилини-годину — для фонової задачі з прогресом це ок; якщо латентність
  батчу дратуватиме, звичайні виклики як фолбек (удвічі дорожче).
- Облік у `storage.record_ai_usage` як усі AI-виклики (/aicost).
- **Підписка Claude Max ≠ API:** токени підписки НЕ можна використати для
  бота (окремі гаманці; хаки з OAuth-токенами Claude Code у своїх скриптах —
  порушення умов, ризик блокування акаунта, не робити). Легальний гібрид для
  ОСОБИСТИХ розкопок Олега: сесія Claude Code під Max + доступ до нори через
  публічний URL Railway Postgres (DATABASE_PUBLIC_URL) — map-reduce робить
  сам Claude Code інтерактивно, токени з підписки. Це «верстат для Олега»
  (вручну, тільки його акаунт, їсть ліміти Max), а /dossier_deep у боті —
  продукт для всієї редакції через API.

### 2.3 З сутнісним шаром (§3) глибоке досьє стає дешевим назавжди

Map робиться ОДИН раз для всього архіву (batch), досьє = SQL-вибірка фактів
по сутності + один reduce. Яруси 2.2 і 3 — одна ідея з різною амортизацією:
2.2 платить ~$1/досьє одразу, 3 платить ~$250–400 раз і далі копійки.

## 3. Сутнісний шар (хвиля D в `ARCHIVE_INTELLIGENCE.md`)

### 3.1 Що витягаємо з кожної статті

Мало типів `kind` (стабільність витягу — вся цінність шару в консистентності),
гранулярність через `subtype`. Кожну статтю модель читає РАЗ і віддає всі типи
одним JSON (див. §3.3 — чому не по одному типу).

| kind | що | subtype (приклади) |
|---|---|---|
| `person` | люди: політики, чиновники, депутати, бізнес, силовики, фігуранти, потерпілі | — |
| `org` | організації | gov, company, party, media, ngo, law_enforcement, utility |
| `place` | географія **+ названі фізичні об'єкти** | district, street, settlement, facility |
| `document` | офіційні акти (лінк до `documents.py`/`prozorro.py`) | decision, order, tender, court, law |
| `event` | названі знакові події (вузько!) | election, attack, disaster |

**Правила таксономії:**
- **`place` зливає географію й об'єкти** (стадіон «Центральний», міст, парк, порт,
  завод-як-будівля). Стадіон — зв'язуюча сутність сценарію «ремонт 2026 ↔ скандал
  2014», тому це повноцінна сутність. Окремий `object` дав би розмиту межу
  (парк — location чи object?), тому один `place` з `subtype=facility`.
- **`event` найрозмитіший** (нормалізація «сесія 25.12» — сутність чи екземпляр?):
  брати лише названі знакові події; процедурну рутину (кожна сесія) — НЕ як сутність.
- **`kind` мінімальний, деталі — в `subtype`:** не плодимо типи (company/party/media
  окремими kind), інакше модель плутає межі й падає консистентність. 5 типів для
  витягу, гранулярність атрибутом: «усі підприємства» = `kind=org AND subtype=company`.
- **Атрибути кожної згадки (не окремі kind):** роль на момент, організація,
  дати/суми/цитати — це ФАКТИ в тексті, не сутності. Витягуються в map-фазі досьє,
  не як entity.

**Що свідомо НЕ беремо:**
- `topic`/тему (тарифи, децентралізація) — **вже покрито тегами** (теги = про що,
  сутності = хто/що конкретне; не дублюємо).
- дати, суми, кількості — факти-атрибути, не сутності для досьє.
- продукти, абстрактні поняття — для локального медіа шум.

Новий `kind` додається лише коли є запит, що без нього не робиться — **АЛЕ** додати
kind ПІСЛЯ прогону = повторно читати весь архів (§3.3), тому таксономію фіксуємо
ПОВНІСТЮ до першого прогону, включно з `event` (слабку якість фільтруємо, але не
переплачуємо за друге читання).

Пізніше (не в першій ітерації, окремий крок): цитати, обіцянки (обіцянкометр).

### 3.1.1 Наочна модель (форма і зміст)

Стаття 2016 (id 80912): «Сенкевич назначил Рохова своим помощником».

**Модель читає текст → віддає JSON:**
```json
{"entities": [
  {"kind":"person","name_ua":"Олександр Сєнкевич","name_ru":"Александр Сенкевич",
   "role":"міський голова Миколаєва","salience":"mentioned"},
  {"kind":"person","name_ua":"Руслан Рохов","name_ru":"Руслан Рохов",
   "role":"помічник міського голови","salience":"main"},
  {"kind":"org","name_ua":"Миколаївська міськрада","name_ru":"мэрия Николаева",
   "salience":"mentioned"}
]}
```

**Лягає у дві таблиці** (та сама форма, що вже обкатано на тегах):

`entities` — картки реальних речей (одна на людину/організацію):
| id | kind | name_ua | name_ru | role_last | mentions |
|--|--|--|--|--|--|
| 1 | person | Олександр Сєнкевич | Александр Сенкевич | мер Миколаєва | 3040 |
| 2 | person | Руслан Рохов | Руслан Рохов | помічник мера | 88 |
| 3 | org | Миколаївська міськрада | мэрия Николаева | — | 15200 |

`article_entities` — зв'язки стаття↔сутність, з роллю і центральністю:
| article_id | entity_id | role_at_time | salience |
|--|--|--|--|
| 80912 | 1 | мер | mentioned |
| 80912 | 2 | помічник мера | **main** |
| 80912 | 3 | — | mentioned |

**Три запити, яких FTS не вміє:**
1. Досьє = `JOIN article_entities WHERE entity_id=1` → ВСІ 3040 статей про Сєнкевича
   (включно з «мер» без прізвища; `salience=main` зверху), а не 24 по слову.
2. Роль у часі: `role_at_time` по роках → траєкторія депутат→секретар→мер.
3. Зв'язки безкоштовно: `сутності, що ділять статті з Вороновим ORDER BY к-сть` →
   міськрада, колеги, забудовники спливають самі (співпоява, звичайний JOIN).

### 3.2 Схема (розширення нори, патерн tags/article_tags)

```sql
entities (
  id BIGSERIAL PK,
  kind TEXT,            -- person | org | place | document | event
  subtype TEXT,         -- деталізація в межах kind (gov/company/facility/decision…)
  name_ua TEXT, name_ru TEXT,
  aliases TEXT[],       -- варіанти написання, що траплялись
  role_last TEXT,       -- остання відома роль (для показу)
  first_seen BIGINT, last_seen BIGINT,  -- unix, межі згадок
  mentions INT          -- лічильник статей
)
article_entities (
  article_id BIGINT, entity_id BIGINT,
  role_at_time TEXT,    -- роль на момент статті («депутат міськради»)
  salience TEXT,        -- main | mentioned (стаття ПРО сутність чи прохідна згадка)
  PRIMARY KEY (article_id, entity_id)
)
```

**Зв'язки явно НЕ моделюємо (анти-скоуп).** «Воронов↔міськрада» виникає зі
співпояви: спільні article_id → SQL-join. Дешево, без галюцинацій зв'язків,
просто запитується. Явний типізований граф — тільки якщо реально впремося.

### 3.3 Пайплайн витягу

- **Batch API** (−50%), **Haiku 4.5**, structured outputs (суворий JSON).
- **ОДИН прохід, УСІ kind одразу — не по одному типу.** Дорога частина —
  читання статті (~1200 вх ткн); вихід малий (~200). Прочитати раз і витягти
  всі 5 типів = 340M вх (~$312). Читати по типу (5 проходів) = 1.7B вх, ~3×
  дорожче за нуль користі — модель п'ять разів перечитує той самий текст.
  Витягти person+org+place+document+event одним JSON модель вміє добре.
- **Наслідок для планування:** додати `kind` ПІЗНІШЕ = повторно прочитати весь
  архів (повна ціна). Тому в перший прохід кладемо ВСІ типи (§3.1), навіть event.
- **Фазування — по РОКАХ, не по типах:** all-kinds на 2020→2026 (найзапитаніші),
  потім углиб до 2009. Кожна стаття читається раз, вартість розтягнута в часі.
  Обсяг: ~286к статей × ~1.2к ткн ≈ 340M вх → ~$170 вх батчем + вихід
  ~$50–80 → **бюджет $250–400 на весь архів**, платиться частинами.
- **Злиття сутностей:** нормалізоване ім'я + алiаси; точний збіг → одна
  сутність. **Однофамільці — головний ризик:** «Воронов» може бути не один.
  Правило: не зливати без збігу ролі/організації/періоду; сумнівні лишати
  окремими сутностями; ручне злиття командою `/entity_merge <id1> <id2>`
  (людський апрув у чаті). Пізніше — embeddings для fuzzy-злиття (хвиля B).
- **Інкремент:** нові статті проходять витяг у щогодинному sync (:50),
  ~$0.001/стаття — непомітно.
- Витяг ролей: зберігати тільки текстуально підтверджене; сумнівне — не писати.

### 3.3.1 Виконання прогону: через Claude Code під Max замість API Batch

Дві легальні доріжки зробити той самий one-time прогін §3.3:

**А. API Batch (нативно, платно).** Скрипт бота ганяє Haiku 4.5 через Batch
API, −50%, ~$250–400 разово, весь архів за години. Найшвидше й найпростіше в
експлуатації, але це живі гроші.

**Б. Через Claude Code під Max (ОБРАНО, безкоштовно з підписки).** Інтерактивна
сесія Claude Code (гаманець Max, не API) конектиться до нори публічним URL
(`DATABASE_PUBLIC_URL`), тягне статті партіями, витяг робить Haiku —
суб-агентами на Haiku (`Agent tool, model=haiku`), — і пише
`entities`/`article_entities` назад тим самим конектом. Легально: це
інтерактивне користування Claude Code над власними даними (не перепродаж API,
не OAuth-хаки).

Тонкощі Б, які треба закласти:
- **Повільно й партіями.** Ліміти Max (5-год вікно + тижневий cap) = не години,
  а тижні сесій. Тому обов'язковий **resumable-курсор** `entity_last_id` у
  `sync_state` (той самий патерн, що бекфіл архіву): сесія жує скільки встигне
  до тротлінгу, наступна продовжує з місця.
- **Haiku, не Opus.** Витяг делегувати суб-агентам Haiku (дешево на лімітах Max
  і саме та модель, що в проді робитиме інкремент). Головна сесія лише
  оркеструє: тягне пачку, роздає агентам, звіряє JSON, пише в БД.
- **Пише в нору напряму.** На відміну від БД сайту (read-only) — нора це власна
  БД бота, запис у неї штатний. Для цього прогону публічний URL береться з
  правом запису (INSERT/UPDATE у `entities`/`article_entities`), а не через
  read-only `/nora_sql`.
- **Спершу тест, потім масштаб.** ~50–150 статей у тестовому режимі →
  перевірка критеріїв §3.6 (точність ≥90%, вигадані ролі ≤2%) → лише тоді
  курсор-цикл на весь архів, фазами по роках (2020→2026→углиб, §3.3).
- **Якість = та сама модель.** Прогін Haiku валідує саме те, що піде в
  прод-інкремент; безкоштовний тест на «іншій моделі» цього не дав би.

Компроміс: Б економить $250–400 ціною тижнів фонової роботи через інтерфейс.
А лишається фолбеком, якщо треба швидко або набридне тягти сесіями. Інкремент
нових статей у проді (§3.3, щогодинний sync) все одно йде через API — Max-канал
лише для разового бекфілу архіву.

### 3.4 Що відкривається

- Досьє на персону збирає **всі** статті через `article_entities` (не
  текстовий пошук), `salience=main` зверху; згадка «Виталий» без прізвища —
  знаходиться.
- NLQ-питання нового класу: «хто найчастіше фігурував з міськрадою у 2015?»,
  «з ким пов'язаний забудовник X?» (співпоява), «усі депутати, згадані з
  тендерами» (kind+співпоява).
- Проактивний контекст (хвиля C) матчить нову статтю зі старими по сутностях,
  а не по словах — значно точніше.
- Досьє персон: «Воронов: перша згадка 2010, ролі за роками, ключові епізоди».

### 3.6 Критерії приймання

- `/dossier Воронов` бачить ~647 матеріалів (усі згадки), main-статті зверху.
- Стаття, де персона лише в тілі тексту (не в заголовку) — у досьє.
- Ручна перевірка 50 випадкових статей: точність сутностей ≥90%, вигаданих
  ролей ≤2%.
