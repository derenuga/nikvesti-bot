## 0. Знахідки ревізії (баги і технічний борг)

### 0.1 `check_channel_silence` — зсув таймзони і відсутність cooldown

`handlers/scheduler.py:77` — перевірка «мовчання каналу» використовує `datetime.now()`,
а Railway працює на UTC. Умова `now.hour < 10 or now.hour >= 18` означає, що вікно
фактично 13:00–21:00 за Києвом, а не 10:00–18:00, як задокументовано. Те саме з
`weekday()`. Крім того, немає cooldown: якщо канал мовчить, нагадування летітиме
**кожні 30 хвилин** до першого поста.

Фікс: рахувати час через `zoneinfo.ZoneInfo("Europe/Kiev")` + запам'ятовувати час
останнього нагадування (не частіше ніж раз на 2 години).

### 0.2 `storage.py` — задубльовані функції

219–241), між ними завис службовий коментар «Ці дві функції потрібно ДОДАТИ...» —
артефакт workflow «повні файли». Python мовчки бере друге визначення, шкоди немає,

## а) Розвиток — пропозиції на апрув

Відсортовано за співвідношенням цінність/зусилля.

### а.1 Пам'ять діалогу для NLQ (multi-turn) — головна пропозиція

Зараз кожне повідомлення Лису — ізольований one-shot: «а за минулий місяць?» після
питання про трафік не спрацює. Тримати в пам'яті процесу короткий контекст
(останні ~6 обмінів на chat_id, TTL ~30 хв) і передавати в `messages`. Це
найбільший стрибок «розумності» за найменші гроші — інфраструктури не треба взагалі.

### а.2 Tools над станом моніторингу (storage) для NLQ

У `/data/prozorro_state.json` вже лежить історія: тендери з сумами, замовниками,
хто взяв у роботу; бачені документи. Додати 2–3 read-only tools
(`get_recent_tenders(period, min_amount?)`, `get_tender_stats(period)`) — і Лис
відповідатиме на «що там по тендерах за тиждень?», «скільки тендерів взяли в
роботу цього місяця?». Дані вже є, потрібні лише обгортки.

### а.3 Meta-tools для NLQ (з беклогу NLQ-модуля)

`get_facebook_top_posts`, `get_instagram_stats` — обгортки навколо існуючого коду
`facebook.py`/`instagram.py`. Закриває питання типу «як зайшов учорашній пост про X».

### а.4 Проактивний Лис: детектор трафікових аномалій

Раз на годину порівнювати трафік поточної години (GA4) з типовим профілем цієї
години/дня тижня. Сплеск ×2–3 → Лис сам пише в чат: «стаття X залетіла — схоже на
Discover» (+ перевірка через Search Console `search_type='discover'`). Для редакції
це пряма редакційна цінність: вчасно підхопити тему, що полетіла, доки вона гаряча.

### а.5 «Тижневик Лиса» — понеділок 09:30

Тиждень у цифрах одним постом: GA4 (тиждень до тижня), топ-5 матеріалів, соцмережі,
скільки тендерів/документів винюхав, найактивніший конкурент — з AI-коментарем.
Весь код збору вже існує в різних модулях, це агрегація + один промпт.

### а.6 Пошук по сайту без очікування KEY4

Проміжний крок до кластеру 3 (LONG_TERM_VISION): tool `search_site_articles` поверх
публічного пошуку nikvesti.com або sitemap (парсинг, як уже робимо для конкурентів).
Не RAG по 17 роках, але «що ми останнє писали про Сєнкевича?» почне працювати вже
зараз, без сервера і БД.

**Примітка після обговорення (01.07.2026):** слушне заперечення Олега — sitemap
величезний і не містить заголовків, пошук по ньому не працюватиме. Реальні варіанти:
(1) публічна сторінка пошуку сайту, якщо вона є і віддає результати без JS —
парсити її як конкурентів; (2) Google Custom Search API обмежений на site:nikvesti.com
(100 запитів/день безкоштовно — для внутрішнього бота цього достатньо). Перевірити
обидва перед кодингом. Якщо прямий доступ до MySQL з'явиться раніше (див. розділ
про підключення до БД) — цей пункт стає непотрібним, робити одразу поверх БД.

### а.7 Контроль якості заголовків (беклог п. 14–15 — можна вже)

Нові матеріали з RSS/sitemap → Haiku перевіряє заголовок і лід на одруківки/росіянізми
→ якщо є що виправити, тихий пост у чат. Дешево (Haiku), не потребує БД.

### Що свідомо НЕ пропоную зараз

Монетизація/підписки і міграція на KEY4+MySQL — за правилами `LONG_TERM_VISION.md`
чекають окремого явного «кодь». Пункти а.1–а.7 навмисно підібрані так, щоб не
залежати від цієї інфраструктури.

## б) Оптимізація існуючого

### б.1 Блокування event loop — головний технічний пункт

Синхронні виклики в async-контексті блокують **весь бот**:

- `client.messages.create` (sync Anthropic SDK) в `ai_messages.py`, `morning.py`,
  `scheduler.py`, `query_router.py`
- tool-функції NLQ (`result = func(**block.input)` в `query_router.py:679`) — GA4,
  Search Console, HTTP-запити `get_author_from_url` виконуються прямо в event loop
- `requests.get` в `morning.py`, `events.py`, `facebook.py`, `instagram.py`, `stat.py`

Поки Лис думає над NLQ (до 4 ітерацій × Claude API + GA4), бот не реагує ні на
команди, ні на реакції. Показово: `documents.py`, `competitors.py`, `prozorro.py`,
`law_enforcement.py` вже роблять це правильно через `run_in_executor` — друга
половина модулів ні.

Фікс: `anthropic.AsyncAnthropic` для всіх AI-викликів + `asyncio.to_thread(...)`
для tool-функцій і requests. Найбільший виграш відчутності за один вечір роботи.

### б.2 Prompt caching в query_router

Масив `TOOLS` (~3–4 тис. токенів) + системний промпт пересилаються з **кожною**
ітерацією tool use (до 4 на запит). `cache_control: {"type": "ephemeral"}` на
останньому tool і на system-блоці → ~80–90% економії input-токенів NLQ. Це і є
реально корисна частина беклог-пункту 1 «AI рефакторинг» (виносити сам
FOX_SYSTEM_PROMPT у кеш сенсу мало — він ~700 символів).

### б.3 Стратегія моделей за вартістю

Зараз усе на `claude-sonnet-4-6`. Пропозиція:
- короткі «смакові» тексти (ДН, нагадування про пошту, підводки соцзвітів) →
  **Haiku 4.5** — у рази дешевше і швидше, на 2–4 реченнях різниці не видно
- NLQ-роутер → **Sonnet 5** (там різниця в якості реально відчутна: вибір tools,
  інтерпретація даних)
- `anthropic==0.40.0` у requirements.txt застарів — оновити SDK

### б.4 Ріст storage без обмежень

Списки seen ids (документи, конкуренти) і словник `tenders`/`message_to_tender`
ніколи не чистяться — JSON росте вічно, а кожна операція = повний read+write файлу
під глобальним lock. Обрізати seen-списки до останніх ~500 на джерело, тендери
старші ~90 днів прюнити (реакції на старі повідомлення все одно не актуальні).

### б.5 Помилки видно тільки в Railway logs

Всі `except` роблять `print(...)`. Якщо вночі відвалиться парсер міськради — ніхто
не дізнається, доки не помітять тишу. Додати error-notifier: виняток у scheduled job →
коротке повідомлення в особистий чат Олегу (з rate limit, щоб не заспамити при
масовому падінні). Заодно перейти з `print` на `logging`.

### б.6 Дрібниці

- `_escape_html` продубльована в `morning.py`, `competitors.py`, `documents.py` → в `helpers.py`
- `morning.py:198`: `strftime('%A')` дає англійську назву дня («Monday») у промпті —
  передавати українську
- `last_channel_post_time` скидається на `now()` при кожному рестарті — після деплою
  перші 2 години тиші не детектяться (прийнятно, але варто знати)
- group-reply leak з NLQ-доки: додати логування user_id/chat_id на вході в
  `group_reply_to_bot`, щоб при повторенні інциденту була фактура

## в) AI-шар: як зробити Лиса розумнішим

Пункти а.1–а.4 — теж про AI-шар; тут решта, специфічна для архітектури.

### в.1 Єдина точка AI-викликів

Зараз `anthropic.Anthropic(...)` створюється в 4 місцях, виклик `messages.create`
скопійований ~10 разів. Один хелпер `fox_generate(prompt, *, mode_prompt=None,
model=..., max_tokens=...)` в `ai_messages.py` → зміна моделі, caching, retry,
логування вартості робляться в одному місці. Передумова для б.2–б.3.

### в.2 MAX_TOOL_ITERATIONS: 4 → 8

Складне питання («порівняй трафік за два місяці по джерелах і намалюй графік») —
це вже 4+ викликів tools, і зараз воно обрізається повідомленням «забагато кроків».
З prompt caching (б.2) додаткові ітерації майже безкоштовні.

### в.3 Живий прогрес у плейсхолдері

Замість статичного «Розбираюсь з вашим питанням, шефе...» — редагувати плейсхолдер
на кожній ітерації: «🦊 Дивлюсь GA4 за вчора…», «🦊 Звіряю з Search Console…»
(мапа tool → людський опис). Дешевий, але дуже відчутний UX-він для запитів на 10+ секунд.

### в.4 Контекст команди в NLQ

Передати роутеру стислий witness `TEAM` (хто є хто, без ДН) — щоб «як зайшли статті
Аліси за тиждень?» коректно атрибутувалось через автора з `get_ga4_top_articles`.

### в.5 Дисципліна вартості

Логувати `usage` (input/output tokens) кожного NLQ-запиту в storage → раз на місяць
Лис сам звітує скільки він коштував. Тримає чесним і покаже ефект від б.2/б.3.
