# Бюджет Миколаєва в норі: версії плану і рішення про зміни

Модулі: `handlers/budget_revisions.py` (задание №2 — ревізії плану з рішень
сесії) і `handlers/budget_snapshots.py` (задание №1 — місячні снапшоти
виконання). Обидва живуть у схемі `budget` Postgres бота (нора), існуючих
таблиць не чіпають.

## UX: кинув ZIP — і все

Редакція отримує пакети рішень сесії від депфіну (тека з PDF рішення,
PDF-додатками і xlsx «Порівняльними таблицями»). Робочий сценарій: **заархівуй
теку пакета і кинь ZIP боту в приват** — без команд і аргументів. Бот сам:

1. розпаковує (лагодить мак-імена cp437→utf-8), знаходить усі xlsx;
2. визначає тип таблиці **за шапкою всередині файлу** — імена файлів брешуть
   (у s-fi-003 файл «Додаток 5» містив таблицю до додатка 3, а «Додаток 1» і
   «Додаток 2» були байт-у-байт одним файлом); дублі відсіює по md5;
3. вантажить порівняльні таблиці додатків 1 (доходи) і 3 (видатки); решту
   (2, 4, 5, 6, 7) пропускає з позначкою;
4. рік і базове рішення (50/26 від 30.12.2025) читає з шапок; код пакета
   (s-fi-XXX) — з імен файлів; порядок нашаровування = порядок надсилання;
5. заголовні показники (доходи/видатки/фонди/резервний фонд) тягне з
   PDF самого рішення (текст цифровий) у `plan_headline`;
6. **нульовий пакет** (вихідний бюджет, все в PDF, xlsx немає) → створює
   original-ревізію без рядків + headline з тексту рішення; рядки доллються
   з лівої частини першої порівняльної таблиці, а номер/дата рішення
   канонізуються з її шапки. Парсер PDF-додатків — план на бюджет-2027;
7. повторне надсилання того ж пакета ревізію не дублює (рядки й issues
   перезаписуються).

## Модель

«Викласти додаток у новій редакції» = повна заміна без дельт. Порівняльна
таблиця: лівий блок «Чинна редакція», правий «Запропонована редакція з
урахуванням змін», порядково вирівняні. Права частина → нова ревізія
(`plan_revision` + рядки), ліва → **валідація** проти останньої збереженої
ревізії року; розбіжності пишуться в `revision_validation_issue` і НЕ
зупиняють завантаження.

Розбіжності — реальність двох сортів (обидва впіймані на живих пакетах 2026):
- **міжсесійні зміни** (розпорядження мера по трансфертах, виконком по
  постанові КМУ №18): «чинна редакція» квітневої таблиці ≠ права частина
  березневої на 49 позицій — включно з виділенням програми 0217427 із
  0217413 і новими 0712170/1014083. Снапшот Казначейства на 01.07
  (7 077,9 млн) теж вище липневої редакції рішень — план живе між сесіями;
- **одруківки депфіну**: у s-fi-001 «чинна редакція» мала різне розбиття
  споживання/розвиток у парі 0200000/0210000 (ловиться перевіркою пар
  «головний розпорядник ↔ відповідальний виконавець», поле `pair:*`);
  у s-fi-003 рядки 0712170/1014083 загублені з таблиці, але у підсумку
  «УСЬОГО» враховані (ловиться контролем сум, `total_match=False`).

Тому сума дельт рішень НЕ зобов'язана пояснювати всю динаміку плану — план
із місячних снапшотів (задание №1) буде первинним фактом, рішення дають
атрибуцію.

## Таблиці (схема budget)

| Таблиця | Що |
|---|---|
| `plan_revision` | Реєстр редакцій: рік, номер рішення (або код s-fi-XXX до ухвалення), дата, kind original/amendment, effective_order (0 = original), UNIQUE(рік, order) |
| `plan_expenditure_line` | Видатки (Додаток 3): КПКВК/ТПКВК/КФК + 12 грошових колонок, `is_unit_total` для підсумків розпорядників (XX00000/XX10000), UNIQUE(revision, kpkvk) |
| `plan_revenue_line` | Доходи (Додаток 1): код (8 зн., ієрархічний — групи X0000000 лежать поруч із листовими), total/general_fund/special_fund/special_dev_budget, UNIQUE(revision, code) |
| `plan_headline` | Показники текстової частини рішення — авто з PDF + ручні поправки |
| `revision_validation_issue` | Розбіжності: чинна vs збережена (`presence`/поле) і пари розпорядник/виконавець (`pair:*`) |

В'юхи: `v_plan_amendments` (дельти видатків між ревізіями; попередник — остання
ревізія року **з видатковими рядками**, бо рішення могло не чіпати додаток;
без unit totals; прапор нових програм), `v_plan_revenue_amendments`,
`v_plan_disappeared` (зниклі програми).

Ревізія без дохідної таблиці (квітень/липень 2026 міняли лише видатки) —
норма: доходи успадковуються, в'юхи це враховують.

## Парсер (перевірено на живих пакетах s-fi-022/001/003/005/007 + s-fi-005/2025)

- Межа блоків — **повторний старт нумерації колонок з «1»** («1…16 | 1…16»
  видатки, «1…6 | 1…6» доходи), не фіксовані номери колонок.
- Кінець таблиці — ТІЛЬКИ рядок з кодом «Х» («УСЬОГО» / «Разом доходів»):
  у дохідній таблиці посеред даних стоять проміжні «Усього доходів (без
  урахування…)» без коду — вони не обривають парсинг.
- Коди — числа без ведучих нулів: КПКВК 7, ТПКВК 4, КФК 4, доходи 8;
  1.4549E9 → '1454900000'.
- Контроль сум: видатки — сума програмних рядків (без unit totals) proti «Х»;
  доходи — сума верхнього рівня ієрархії (X0000000).
- Нова програма/рядок = порожній лівий блок; у валідацію не пише.

## Команди

| Команда | Що |
|---|---|
| *(без команди)* ZIP/xlsx у приват | Основний шлях — пакет розбирається автоматично |
| `/budget_load [рік] [номер] [дата]` | Ручний шлях для одного файлу (підпис або reply), аргументи опційні |
| `/budget_status` | Ревізії по роках: рядки, суми, розбіжності |
| `/budget_headline <рік> <номер> ключ=значення …` | Поправити заголовні показники вручну |

Доступ: `ALLOWED_USER_IDS`. Без `BOT_DATABASE_URL` модуль неактивний.

## Задание №1: місячні снапшоти виконання (budget_snapshots.py)

Джерело — «Щомісячна інформація» mkrada
(`/content/shchomisyachna-informaciya-{РІК}r.html`): пари xlsx «Витрати…» /
«Надходження… DD.MM.YYYY.xlsx». Обрано САЙТ, а не OpenBudget API, бо тільки
сайт дає розріз по розпорядниках (КВК) — «який чиновник не виконав план у
термін», головна редакційна цінність.

- Таблиці: `snapshot` (рік+дата+kind), `snapshot_expenditure_line`
  (розпорядники `code_type='unit'` + деталізація КЕКВ (Лист1) і КБП (Лист2);
  блок після «Разом» = весь бюджет, синтетичний КВК '00'; рядок «Разом» —
  `code_type='total'`), `snapshot_revenue_line` (назви по фондах, кодів у
  файлі немає; `fund='total'` — «Всього доходів»).
- Дата зрізу — з імені файлу (у шапках її немає); фіскальний рік =
  (дата − 1 день).рік — файл за 01.01.2027 лягає у 2026-й.
- Монітор: щодня об 11:20; перший запуск бекфілить усю історію року тихо,
  звіт шле лише за останній місяць (baseline-правило). Стан — sync_state
  `budget_snapshots_seen`. Сторінка віддає кириличні href — форсуємо utf-8.
- Реальні граблі: у лютневому файлі загальноміський КЕКВ-блок продубльований
  повністю (значення ідентичні) — дедуп «перший виграв».
- `/budget_execution`: розпорядники з виконанням плану періоду <80%
  (на 01.07.2026: УКБ 13,3%, депфін 31,7%…) + звірка річного плану знімка з
  останньою ревізією рішень. Методологія знімка «без кредитів і власних
  надходжень установ», тому різниця = міжсесійні зміни ± методологія —
  подається як сигнал, не як помилка.

## OpenBudget (незалежна звірка)

`api.openbudget.gov.ua/api/public/localBudgetData?budgetCode=1454900000&budgetItem=EXPENSES&classificationType=PROGRAM&period=MONTH&year=2026`
→ CSV (windows-1251): затверджено/план/касові по місяцях, фонд × ТПКВК (4 зн.)
× КЕКВ. Є INCOMES по кодах доходів. Ревізії рішень туди НЕ приходять —
казначейська звітність уже «розчинена», без атрибуції і без розбивки по
розпорядниках. Снапшоти сайту міськради (щомісячна інформація) — розріз
КВК×КЕКВ/функції і доходи за назвами, гірше за API. Swagger:
`api.openbudget.gov.ua/swagger-ui.html` (docs: confluence-ext.spending.gov.ua).
Прямий доступ з дата-центрів Akamai ріже — з Railway перевірити окремо.
