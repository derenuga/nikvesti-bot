## Основний API (активний графік)

### Отримати черги за типом
```
GET /api/outage-queue/by-type/{type_id}
```
Повертає масив: `[{"id": int, "name": str, "enabled": int, ...}]`  
`name` — людська назва черги, наприклад `"2"`, `"2.2"`, `"4.1"`.

### Активний розклад
```
GET /api/v2/schedule/active
```
Повертає масив активних розкладів. Кожен має поля `from`, `to` (ISO 8601, UTC), і `series` — масив слотів.

Структура слоту в `series`:
```json
{"outage_queue_id": int, "time_series_id": int, "type": "OFF"|"SURE_OFF"|"PROBABLY_OFF"}
```
- `time_series_id` — 30-хвилинний слот (1 = 00:00–00:30, 48 = 23:30–00:00).
- `type`: `OFF`/`SURE_OFF` → точне відключення; `PROBABLY_OFF` → можливе.

### Функції в коді
- `fetch_outage_data()` → `{"queues": dict, "active": list}` — один виклик, збирає все потрібне.
- `build_message(data)` → готовий текст повідомлення українською.
- `_find_current_schedule(active)` → повертає поточний або найближчий майбутній розклад.
- `_merge_slots(slot_ids)` → зливає суміжні 30-хв слоти в діапазони `("HH:MM", "HH:MM")`.

## Адресний API (каскад для CSV-експорту)

Повна ієрархія: **дільниця → населений пункт → вулиця → будинок → черга**.

### Ендпоінти (всі підтверджені робочі)

```
GET /api/addr/filii
```
Повертає всі дільниці: `[{"idfilial": int, "fullname": str}, ...]`  
⚠️ Ключ — `idfilial`, не `id`.

```
GET /api/addr/filii/{idfilial}/ns
```
Населені пункти дільниці: `[{"idnaspunkt": int, "naznaspunkt": str}, ...]`

```
GET /api/addr/ns/{idnaspunkt}/street
```
Вулиці населеного пункту: `[{"idstreet": int, "nazstreet": str}, ...]`

```
GET /api/addr/street/{idstreet}/dom
```
Будинки вулиці: `[{"iddom": int, "nazdom": str}, ...]`

```
GET /api/addr/dom/{iddom}/outage-queue
```
Черги для будинку:
```json
[
  {
    "outage": {"id": int, "name": str, "type_id": int, "enabled": int, "created_at": null, "updated_at": str, "deleted": int},
    "outageType": {"id": int, "name": str, "created_at": null, "updated_at": null}
  }
]
```
`outage.name` — номер черги (`"2"`, `"2.2"`, `"4.1"`).  
`outageType.name` — тип (`"ГАВ"`, `"СГАВ"`, `"ГПВ"`).

### Мертві ендпоінти (404, не використовувати)
- `GET /api/addr/ns` (будь-який query string)
- `GET /api/addr/ns/{id}` (пряма адресація)
- `GET /api/addr/street/{id}/outage-queue` (рівень вулиці — не існує, потрібен будинок)

## Дільниця 15 — Миколаївський РЕМ (місто Миколаїв)

`idfilial = 15` — основна дільниця для Миколаєва.  
Всі вулиці міста належать до `naspunkt_id = 171` ("м. Миколаїв").  
CSV-експорт: **736 вулиць** (результат запуску `/outage_export 15` від 2026-06-30).

### Ключовий факт: одна вулиця може входити в кілька черг одночасно
Підтверджено на реальних даних: наприклад, вул. Маріупольська, Шевченка, В. Благовісного → черга `"2"` (СГАВ) + черга `"2.2"` (ГПВ) одночасно. Це нормально і не є помилкою даних — різні типи відключень можуть накладатись.

## CSV-експорт (`/outage_export`)

Команда `/outage_export [idfilial]` (за замовчуванням `idfilial=15`) обходить весь адресний каскад і повертає CSV.

### Архітектурне рішення: семплінг першого будинку на вулиці
Черга призначається не на будинок, а на вулицю (або район) — перший будинок зі списку репрезентує всю вулицю. Це зменшує кількість HTTP-запитів з десятків тисяч до сотень (~2 запити на вулицю замість 2 × кількість будинків).

### Структура CSV
```
filiya_id, filiya_name, naspunkt_id, naspunkt_name, street_id, street_name,
sample_dom_id, sample_dom_name, outage_queue_raw
```
`outage_queue_raw` — JSON-рядок з масивом об'єктів `{"outage": {...}, "outageType": {...}}` або `""` (вулиця без будинків) або `"ERROR: ..."`.

### Увага: команда виконується ~7–15 хвилин
Синхронний каскадний обхід без конкурентності. Немає проміжного прогрес-логування (відомий недолік — TODO). Не перезапускати Railway поки не минуло 20 хвилин після запуску.

## Команда `/outage_probe`

Службова розвідувальна команда для тестування ендпоінтів API з Railway (де є мережевий доступ).

```
/outage_probe api/<шлях> [query string]   — запит до API
/outage_probe js/app.js <підрядок>        — пошук в JS-бандлі сайту
```
