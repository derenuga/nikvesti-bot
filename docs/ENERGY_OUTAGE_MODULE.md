## Основний API (активний графік)

### Отримати черги за типом
```
GET /api/outage-queue/by-type/{type_id}
```
Повертає масив: `[{"id": int, "name": str, "enabled": int, ...}]`  
`name` — людська назва черги, наприклад `"2"`, `"2.2"`, `"4.1"`.

### Активний розклад
```
GET /api/v2/schedule/active
```
Повертає масив активних розкладів. Кожен має поля `from`, `to`, `updated_at` (ISO 8601, UTC), і `series` — масив слотів.

Структура слоту в `series`:
```json
{"outage_queue_id": int, "time_series_id": int, "type": "OFF"|"SURE_OFF"|"PROBABLY_OFF"}
```
- `time_series_id` — 30-хвилинний слот (1 = 00:00–00:30, 48 = 23:30–00:00).
- `type`: `OFF`/`SURE_OFF` → точне відключення; `PROBABLY_OFF` → можливе.

### Функції в коді
- `fetch_outage_data()` → `{"queues": dict, "active": list}` — один виклик, збирає все.
- `_parse_schedule(data)` → `(schedule, queue_map)` де `queue_map: name → {off, prob, type_id, type_name}`.
- `build_message(data, changes=None)` → готовий HTML-текст повідомлення.
- `_find_current_schedule(active)` → повертає поточний або найближчий майбутній розклад.
- `_merge_slots(slot_ids)` → зливає суміжні 30-хв слоти в діапазони `("HH:MM", "HH:MM")`.

## Формат повідомлення `/outage`

```
⚡ Графік відключень на 1 липня 2026 (оновлено 18:30)

📋 ГПВ:
<b>Черга 3.1: немає світла 2 год 30 хв</b>
🕐 19:00–21:30 / можливо — 18:30–19:00 / 21:30–22:00
<blockquote expandable>Центр міста Миколаєва, Баштанська, ...</blockquote>
```

- `parse_mode="HTML"` у Telegram
- Рядок черги + сума годин — **жирний** (`<b>`)
- Часові діапазони — окремий рядок з 🕐
- Мікрорайони/дільниці — `<blockquote expandable>` (TG Bot API 7.4+, автозгортання)
- Час оновлення береться з `schedule["updated_at"]`, конвертується в Europe/Kiev
- Години рахуються тільки по `OFF`/`SURE_OFF` слотах (не `PROBABLY_OFF`)
- Діапазони часу через ` / `

## Моніторинг змін (планувальник)

**Job:** кожні 5 хвилин, `check_outage_changes(bot, debug_chat_id)`  
**Стан:** `/data/outage_monitor_state.json` — зберігає `queue_map` після кожного запуску  
**Debug chat:** особистий чат Олега (перший ID з `ALLOWED_USER_IDS`)

### Логіка
1. Перший запуск (стан порожній) — мовчки зберігає baseline, нічого не надсилає
2. Наступні запуски — порівнює нову `queue_map` зі збереженою
3. Якщо є зміни — надсилає повне повідомлення з блоком `🔄 Зміни у графіку:`

### Типи змін
- `➕ Черга X.X з'явилась у графіку`
- `➖ Черга X.X зникла з графіку`
- `✏️ Черга X.X: новий графік — HH:MM–HH:MM`

### Подальший план
Після налагодження — замінити `debug_chat_id` на `PROZORRO_CHAT_ID` (публічний канал).

## Адресний API (каскад для CSV-експорту)

Повна ієрархія: **дільниця → населений пункт → вулиця → будинок → черга**.

### Ендпоінти (всі підтверджені робочі)

```
GET /api/addr/filii                          — всі дільниці
GET /api/addr/filii/{idfilial}/ns            — населені пункти дільниці
GET /api/addr/ns/{idnaspunkt}/street         — вулиці населеного пункту
GET /api/addr/street/{idstreet}/dom          — будинки вулиці
GET /api/addr/dom/{iddom}/outage-queue       — черги для будинку
```

⚠️ Ключ дільниці — `idfilial`, не `id`.

### Мертві ендпоінти (404, не використовувати)
- `GET /api/addr/ns` (будь-який query string)
- `GET /api/addr/ns/{id}` (пряма адресація)
- `GET /api/addr/street/{id}/outage-queue` (рівень вулиці — не існує, потрібен будинок)

## Дільниця 15 — Миколаївський РЕМ (місто Миколаїв)

`idfilial = 15` — основна дільниця для Миколаєва.  
Всі вулиці міста належать до `naspunkt_id = 171` ("м. Миколаїв").  
CSV-експорт: **736 вулиць** (результат запуску `/outage_export 15` від 2026-06-30).

**Ключовий факт:** одна вулиця може входити в кілька черг одночасно.  
Наприклад, вул. Маріупольська → черга `"2"` (ГАВ) + черга `"2.2"` (ГПВ) одночасно. Це норма — різні типи відключень накладаються.

## Команда `/outage_export`

`/outage_export [idfilial]` (за замовчуванням `idfilial=15`) обходить весь адресний каскад і повертає CSV. Результат кешується в `/data/outage_streets_{idfilial}.json`.

**Архітектура:** семплінг першого будинку на вулиці — зменшує HTTP-запити з десятків тисяч до ~2 на вулицю.  
**Час виконання:** ~7–15 хвилин (синхронний каскадний обхід без конкурентності).

## Команда `/outage_probe`

Службова розвідувальна команда для тестування ендпоінтів API з Railway.

```
/outage_probe api/<шлях> [query string]   — запит до API
/outage_probe js/app.js <підрядок>        — пошук в JS-бандлі сайту
```
