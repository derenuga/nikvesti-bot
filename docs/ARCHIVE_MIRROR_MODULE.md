# «Лисяча нора» (foxhole) — корпус архіву + пошук + /dossier

Хвиля A плану [`ARCHIVE_INTELLIGENCE.md`](ARCHIVE_INTELLIGENCE.md) — інституційна
пам'ять редакції поверх 17-річного архіву. Апрув Олега: «Кодь», 03.07.2026;
збагачення (теги, рубрика, регіон, зважений пошук) і фільтр «тільки
опубліковані» — 04.07.2026. «Лисяча нора» — розмовна назва корпусу (технічна
таблиця лишається `articles`).

## Що потрапляє в нору

**Тільки реально опубліковані матеріали:** `type='news'`, `status=1`, `published`
(unix-час) валідний і **в минулому**. Свідомо НЕ беремо: чернетки (`status=0`),
недатовані (`published=0`), відкладені (`published` у майбутньому). Якщо матеріал
знімають з публікації — інкремент прибирає його з нори. Правило єдине для
бекфілу, інкременту й пошуку: у норі рівно те, що читач бачить на сайті.

**Порожні заглушки не беремо:** якщо після чистки HTML обидві текстові версії
порожні (`text_ua` і `text_ru` — None) — стаття в корпус не йде (`_row_to_tuple`
→ None). Це технічні записи ранніх років (напр. 2001 — самі заголовки типу назв
футбольних клубів, без тіла): для пошуку/досьє марні. Курсор бекфілу все одно
проходить повз них (лічильник «оброблено» ≥ фактично залитих — це очікувано).

## Модулі

| Файл | Що робить |
|---|---|
| `handlers/bot_db.py` | Адаптер власної Postgres-БД бота: схема (articles + generated tsvector, sync_state), upsert, курсори, `ping()` |
| `handlers/archive_mirror.py` | Синхронізація дзеркала: `/archive_backfill` (разова заливка, resumable), інкрементальний sync щогодини о :50, `/archive_status` |
| `handlers/archive_search.py` | Повнотекстовий пошук: tsquery з префіксними лексемами, стратифікація по роках, excerpts; NLQ-tool `search_archive_fulltext` |
| `handlers/dossier.py` | `/dossier <тема>`: Haiku генерує ua/ru варіанти пошуку → пошук по роках → Sonnet складає таймлайн з лінками |

## Архітектура

```
MySQL сайту (read-only, ліміти KEY4)
        │  бекфіл (разово, порціями) + інкремент (щогодини :50)
        ▼
Postgres бота (Railway) — articles: id, дати, title_ua/ru, slug, text_plain,
                          fts (generated tsvector, GIN)
        │
        ├── search_archive_fulltext (NLQ-tool Лиса; кнопки беку працюють)
        └── /dossier — історія питання по роках з лінками
```

**Чому дзеркало:** production-БД read-only (індекс не створиш), 10 000 запитів/год,
5 з'єднань; кожен LIKE по longtext — ризик для сайту. HTML → чистий текст
конвертується один раз при синку.

## Налаштування (Railway)

1. Додати **PostgreSQL** у проект `remarkable-stillness`.
2. У сервісі бота задати `BOT_DATABASE_URL` = референс на `DATABASE_URL`
   Postgres-плагіна (бот також бачить і просто `DATABASE_URL` — фолбек).
3. Задеплоїти, перевірити `/archive_status` (покаже «дзеркало порожнє»).
4. **Спершу зразок:** `/archive_sample` — заллє кілька найстаріших (рос. епоха)
   і найновіших (укр.) статей і покаже, що осіло в базі. Звірити очима: чи
   чистка HTML не з'їла зайвого / не лишила сміття, чи мовні версії лягли в
   правильні поля. За потреби підкрутити `_JUNK_*` у `archive_mirror.py`.
5. **Потім фазами:** `/archive_backfill 5000` заллє порцію на 5000 статей і
   зупиниться (можна стежити за диском Postgres між порціями); повторний виклик
   продовжить з курсора. `/archive_backfill` без числа доллє все решта
   (~1–2 год загалом, порціями по 150 з паузою 1 с — у межах лімітів KEY4).
   Resumable: обрив → повторний виклик з місця зупинки.
   **Обсяг:** `nodes` = ~320 тис. записів; при ~3 КБ тексту на новину це ≈ 1 ГБ
   тексту, з обома мовами й FTS-індексом орієнтовно 2–3 ГБ у Postgres.
6. Після повного завершення дзеркало оновлюється само (щогодини о :50).

Без `BOT_DATABASE_URL` бот працює як раніше: `/dossier` і full-text пошук чемно
відмовляють, NLQ-роутер відкочується на `search_news_archive` (LIKE по заголовках).

## Схема Postgres

```sql
articles (
  id BIGINT PK,                 -- = nodes.id сайту
  published, updated BIGINT,    -- unix-час (секунди); у норі лише published>0 та <=now
  status SMALLINT,              -- у норі лише status=1 (див. «Що потрапляє в нору»)
  own_material SMALLINT,        -- 1 = власний матеріал, 0 = рерайт/агентське (фільтр)
  owner_id BIGINT,              -- → users.id сайту (автор, число)
  title_ua TEXT, title_ru TEXT, -- РОЗДІЛЕНО: title_ru = nodes.title (без суфікса — рос.)
  slug TEXT,                    -- slug_ua || slug; URL = nikvesti.com/news/{slug}
  text_ua TEXT, text_ru TEXT,   -- РОЗДІЛЕНО: чистий текст кожної мови, кап 60 000 симв. кожен
  category TEXT, region INTEGER, -- слаг рубрики / код регіону (фільтри)
  tags_text TEXT,               -- назви тегів статті (ua+ru) — для індексу
  fts tsvector GENERATED ALWAYS AS         -- ЗВАЖЕНИЙ: заголовок(A) > теги(B) > текст(C)
      (setweight(titles,'A') || setweight(tags_text,'B') || setweight(text,'C')) STORED
)
tags (id PK, name_ua, name_ru, name_en, iptc, description)  -- канонічні (мерджі зведені)
article_tags (article_id, tag_id, PK(article_id,tag_id))    -- many-to-many, індекс по tag_id
sync_state (key PK, value)      -- backfill_last_id, backfill_done_at, mirror_cursor
```

**Мовні версії суворо розділені** (вимога Олега, 04.07): до ~2023 матеріали були
лише російською (`nodes.content` / `nodes.title`), потім українською
(`content_ua` / `title_ua`), бувають і обидві. Тому НЕ звалюємо в одне поле —
`text_ua`/`text_ru` окремо, нічого не губиться і не змішується. Пошуковий індекс
об'єднує всі чотири поля, тож `search_archive_fulltext` знаходить незалежно від
мови; excerpts для досьє беруть `coalesce(text_ua, text_ru)` (укр. пріоритет).

**Зважений пошук:** `fts` — generated tsvector зі `setweight`: заголовок = A,
теги = B, текст = C. `ts_rank` за замовчуванням A=1.0 > B=0.4 > C=0.2 — збіг у
заголовку/тезі ранжується вище за випадкову згадку в тілі.

### Теги, рубрика, регіон (збагачення 04.07)

- `category` (слаг рубрики), `region` (числовий код) — колонки статті; фільтри
  в пошуку. Значення зберігаємо сирими, зміст (число→регіон, перелік слагів)
  інтерпретуємо при показі; мапінг Олег дасть потім.
- **Теги** — три складові:
  1. `tags` — довідник (дзеркало `tag`): `name_ua/ru/en`, `iptc` (= сайтова
     `google_category`, IPTC Media Topics), `description`. **Мерджені теги
     зведені**: `redirect_tag_id` ланцюгом розв'язується до канонічного, у норі
     тільки канонічні.
  2. `article_tags` — зв'язок many-to-many (з `node_tag`), для «всі матеріали з
     тегом X», спільних тегів, аналітики за IPTC.
  3. `tags_text` — назви тегів статті (ua+ru) денормалізовано в рядок і **вкладено
     в пошуковий індекс** (вага B): тег «нерухомість» знайдеться, навіть якщо
     слова немає в тексті.
- Теги наслоювались із роками — у старих (2008–2011) матеріалів їх немає, це
  нормально (`tags_text` NULL). Довідник `tag` тягнемо раз на прогін, `node_tag`
  читаємо пачками (на кожні 150 статей — один запит), не по одній.
- Стійкість: якщо `tag`/`node_tag` недоступні — синк іде без тегів, не падає.
- Ре-синк коректно знімає теги: `replace_article_tags` робить DELETE-then-INSERT
  для пачки, тож знятий на сайті тег зникає і в норі.

**own_material — слабкий/розріджений сигнал:** колонка з'явилась на сайті не
одразу, тож для старих матеріалів машинно не сказати власний vs агентський
(значення просто немає). Фільтр `own_material=true` вмикається ЛИШЕ на явний
запит про власні/оригінальні матеріали; для «що ми писали про X» / історії
питання його НЕ вживаємо — весь сайт це наш контент, і фільтр викинув би
легітимні старі матеріали, що передують колонці (підказка в NLQ-tool це фіксує).

**Автори:** `owner_id` (число) вже пишеться. Дзеркало таблиці `users` (id→ім'я)
і атрибуція «статті автора X» — окремий крок після бекфілу (в.4 ревізії липня).

### Міграції схеми (`SCHEMA_VERSION` у bot_db)

`CREATE TABLE IF NOT EXISTS` не додає нових колонок до вже наявної таблиці — тож
коли з'являються нові колонки, стара нора сама їх НЕ отримає. Тому `ensure_schema`:
1. базові `CREATE ... IF NOT EXISTS` (цільова схема для чистої БД);
2. **ідемпотентні міграції** (`ADD COLUMN IF NOT EXISTS` + залежні індекси) —
   виконуються завжди, доводять стару таблицю до поточної;
3. **версійна перебудова fts** (drop+add generated-колонки — вираз не змінити
   ALTER-ом) лише коли збережена `schema_version` < `SCHEMA_VERSION`.

Індекси рубрики/регіону і fts створюються в кроках 2–3 (не в базі): у старій
таблиці колонок ще немає, і `CREATE INDEX` у базі впав би з "column does not exist".
Піднімати `SCHEMA_VERSION` при кожній зміні, що потребує перебудови generated-полів.
Перевірено на живому Postgres обидва шляхи: чиста БД і міграція старої (v1→v2).

Індекси: GIN(fts), btree(published DESC), опційно pg_trgm по заголовках
(якщо інстанс дозволяє CREATE EXTENSION; без нього все працює).

## Пошук: як влаштована морфологія без стемера

Українського стемера в Postgres немає → конфіг `'simple'` + **префіксні лексеми**:
кожне слово запиту обрізається до грубої основи (`≥8 симв. → -3`, `≥6 → -2`,
`≥5 → -1`) і шукається як `основ:*`. «Стадіону» → `стаді:*` → знаходить
стадіон/стадіону/стадіоном. Ранжування `ts_rank` піднімає точні збіги.

**Відоме обмеження (перевірено тестом):** ua-запит НЕ знаходить російські тексти
(«стадіону» ≠ «стадиона», бо і/и). Закрито на рівні вище: `/dossier` генерує
ua+ru варіанти через Haiku, NLQ-роутеру наказано повторювати пошук російським
написанням. Семантично це закриє хвиля B (embeddings).

`spread_years=true` — режим «історія питання»: `ROW_NUMBER() OVER (PARTITION BY
рік ORDER BY ts_rank)` ≤ per_year, від давніх до свіжих — історія не тоне під
свіжими новинами.

## /dossier

`/dossier стадіон Центральний` →
1. Haiku (`FOX_MODEL_FAST`): 3–5 пошукових варіантів (ua/ru написання, синоніми).
2. Кожен варіант → пошук зі stratify по роках; злиття: ≤3/рік, ≤22 всього.
3. `get_excerpts` — по 700 симв. тексту на статтю з дзеркала.
4. Sonnet (`FOX_MODEL_SMART`): таймлайн по періодах, фігуранти, відкриті питання.
   Правило чесності беків: тільки факти з наших текстів, кожен факт — з
   HTML-лінком анкором 1–3 слова.

Результат лягає в пам'ять діалогу NLQ → follow-up'и («а що по 2016?») працюють.
Вартість одного досьє ≈ $0.05–0.15 (Haiku варіанти + Sonnet синтез ~20К вх. токенів).

| Команда | Дія |

| `/archive_backfill [N]` | Заливка архіву; N — скільки за цей запуск (фазування), без N — усе решта; фонова, resumable |
| `/archive_status` | Стан дзеркала: скільки статей, діапазон дат, курсори, чи йде бекфіл |
| `/dossier <тема>` | Історія питання з архіву: таймлайн по роках з лінками |

## Розклад

`:50` щогодини — інкрементальний sync: `WHERE GREATEST(updated, published) >= cursor-120`,
ORDER BY зміною, LIMIT 500 (аномальний сплеск добереться наступними запусками).
Курсор — `mirror_cursor` у sync_state, **клампнутий на now** (відкладений пост із
`published` у майбутньому інакше зсунув би курсор туди й пропустив усе до того часу).
Тихо пропускається без env або до бекфілу. Змінені рядки тягне **без фільтра
status**, далі ділить: «живі» (status=1, published у минулому) — upsert; решту
(зняті з публікації, чернетки, відкладені) — `delete_articles` прибирає з нори.

## Тестування (03–04.07.2026)

Інтеграційно на реальному PostgreSQL 16:
- схема з generated-колонкою (зважений tsvector), ідемпотентний upsert;
- **розділення мов**: рос-only / укр-only / двомовна стаття — обидві версії
  зберігаються окремо, FTS знаходить обома мовами;
- **пошук по тілу** (слова немає в заголовку), spread_years, фільтри років;
- **фільтр «тільки опубліковані»**: чернетка (status=0), недатована (published=0),
  відкладена (published у майбутньому) — у нору НЕ потрапляють;
- **зняття з публікації**: інкремент видаляє з нори; **клампінг курсора**: майбутній
  published не отруює курсор;
- **теги**: мерджені зведено до канонічних, `tags_text` у пошуку (тег знаходиться
  без слова в тексті), фільтри own_material/category/tag, ре-синк знімає теги;
- **зважене ранжування**: збіг у заголовку вище за збіг у тілі;
- фазовий бекфіл (порція→порція→завершено, позначка done лише наприкінці,
  ідемпотентність), excerpts, злиття варіантів досьє, sync_state, ping.
- `html_to_text` — на задокументованій структурі content_ua.

⚠️ НЕ тестовано на живих БД (з dev-середовища немає доступу: whitelist по IP
Railway). **Після деплою**: `/archive_status` → `/archive_sample` (звірити чистку,
мови, теги очима) → `/archive_backfill N` фазами → спот-чек `/dossier`.

## Стійкість і ліміти

- Обидві БД недоступні / не налаштовані → всі функції чемно відмовляють, бот живе.
- Збій інкрементального sync → `notify_error` Олегу (як у решти моніторів).
- Бекфіл і інкремент не працюють одночасно (`_backfill_running`).
- Розмір: сотні тисяч статей × ≤60К симв. ≈ одиниці ГБ; Railway Postgres тягне.
