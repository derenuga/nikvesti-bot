# Природномовні запити до Лиса Микити (Agentic Query Layer)

**Статус:** Ідея на майбутнє, ще не в розробці. Цей файл — вхідна точка для Claude Code, коли дійдемо до реалізації.

**Не кодити без явного "кодь" від Олега.** Це стосується і цього модуля.

---

## Проблема

Зараз взаємодія з ботом — це жорсткі команди (`/analytics`, `/stat <url>`, `/facebook` тощо) або повідомлення за розкладом. Якщо Олег чи хтось з редакції хоче запитати щось людською мовою —

> "Слухай, а яка у нас на сайті середньомісячна відвідуваність?"
> "Що ми останнє писали про Сєнкевича?"
> "Скільки залученості було у дописа про тендер на школу минулого тижня — і в Facebook, і на сайті?"

— бот цього не розуміє. Кожен такий запит або ігнорується, або вимагає нової жорстко закодованої команди.

## Мета

Дати Лису Миките здатність:
1. **Зрозуміти** довільне питання живою мовою (укр.)
2. **Визначити**, які дані потрібні для відповіді і з яких джерел
3. **Викликати** потрібні внутрішні функції (GA4, Meta, у майбутньому — БД сайту)
4. **Синтезувати** точну відповідь на основі реальних даних — без вигадування фактів

Це класична задача **tool use / function calling** через Anthropic API — Claude сам обирає, які "інструменти" викликати і з якими параметрами, на основі опису доступних функцій.

---

## Архітектурний принцип

Не один окремий "agentic-режим", а **розширення існуючого AI-шару** (`handlers/ai_messages.py`) новим режимом обробки: коли повідомлення в чаті редакції не відповідає жодній відомій команді і не є частиною іншого flow — воно йде в **диспетчер намірів** (intent router).

```
Повідомлення в чаті
        │
        ▼
Чи це команда (/...) ?──так──▶ існуючий handler команди
        │ні
        ▼
Чи це частина іншого flow (conversation state)?──так──▶ той flow
        │ні
        ▼
   Intent Router (новий модуль: handlers/query_router.py)
        │
        ▼
Claude API виклик з tool use:
  системний промпт + список доступних tools + текст питання
        │
        ▼
Claude повертає tool_use блок(и): яку функцію викликати, з якими аргументами
        │
        ▼
Python виконує відповідну функцію (одну або кілька, послідовно або паралельно)
        │
        ▼
Результат(и) повертаються в Claude другим викликом (tool_result)
        │
        ▼
Claude формулює фінальну відповідь людською мовою (стиль Лиса Микити)
        │
        ▼
Відповідь у Telegram
```

Важливо: **інструменти (tools) — це обгортки над вже існуючими функціями**, не нова логіка збору даних. GA4-функції з `google_analytics.py`, Facebook/Instagram функції з `facebook.py`/`instagram.py` — переговорюються в JSON Schema для tool use, але самі дата-функції лишаються тими самими.

## Категорії інструментів (tools)

### 1. GA4 (вже є основа в `google_analytics.py`)

Потрібно узагальнити наявні функції під параметризовані tools, а не лишати їх прив'язаними до конкретних команд:

- `get_ga4_metric(metric, period, start_date?, end_date?)` — узагальнена метрика (`activeUsers`, `screenPageViews`, `screenPageViewsPerSession`, `newUsers`, `returningUsers` — рахується як `activeUsers - newUsers`, оскільки прямої метрики немає в Data API v1beta) за довільний період
- `get_ga4_top_articles(period, limit?)` — топ статей за переглядами з атрибуцією автора (вже є логіка в `/analytics`)
- `get_ga4_article_stats(url)` — статистика конкретної статті (вже є в `/stat`)
- Singapore виключається з усіх GA4-запитів (бот-трафік) — зберегти це правило і тут

**Розбір періоду з мови** ("середньомісячна", "за останній квартал", "у вересні", "за тиждень") — довірити самому Claude через tool use: параметр `period` приймає або стандартизовані значення (`last_7_days`, `last_30_days`, `this_month`, `last_month`, `this_quarter`), або довільні `start_date`/`end_date` в ISO форматі, які Claude сам порахує з контексту поточної дати.

### 2. Meta (Facebook + Instagram, є основа в `facebook.py`, `instagram.py`)

- `get_facebook_post_stats(period | post_url)` — залученість, охоплення (`post_media_view` — поточна метрика API v25.0)
- `get_instagram_stats(period)` — статистика акаунту
- `get_facebook_top_posts(period, limit?)` — топ дописів за залученістю

### 3. Пошук по сайту (MySQL, майбутнє — після БД-доступу, п. 10 беклогу PROJECT_CONTEXT.md)

Це найскладніший і найцінніший інструмент — дозволить питання типу "що останнє ми писали про Мера Сєнкевича?".

- `search_site_articles(query, limit?, date_from?, date_to?)` — повнотекстовий пошук по статтях МикВісті (Laminas/Zend, сервер KEY4)
  - Реалізація пошуку: MySQL `FULLTEXT` індекс (якщо є/можна додати) або `LIKE` по заголовку + тегам як перший грубий варіант, з можливим переходом на повноцінний пошук пізніше
  - Повертає: заголовок, дату публікації, автора, URL, короткий уривок (snippet)
  - **Залежність:** потребує статичного доступу до БД — або після міграції на KEY4 (де бот матиме прямий доступ), або через REST API wrapper на сервері сайту, або SSH tunnel — варіанти вже зафіксовані в PROJECT_CONTEXT.md backlog
- `get_article_by_url(url)` — повний текст/метадані конкретної статті (може бути корисним для крос-перевірки з GA4 `/stat`)

**Важливо для якості відповідей:** для query типу "що ми писали про X" Claude має явно показувати джерело (дата + посилання), а не переказувати вміст своїми словами без посилання на конкретну статтю — щоб редакція могла перевірити.

### 4. (Опційно, пізніше) Комбіновані інструменти

Деякі питання вимагають об'єднання кількох джерел в одному запиті — наприклад "скільки залучення зібрав допис про тендер на школу — і на сайті, і в Facebook". Це не окремий tool, а природний результат того, що Claude сам викликає `search_site_articles` → знаходить статтю → бере її URL → викликає `get_ga4_article_stats(url)` і `get_facebook_post_stats(post_url)` — послідовно, в межах одного diалогу tool use.

## Технічна реалізація (tool use в Anthropic API)

Кожен інструмент описується в форматі JSON Schema для параметра `tools` у виклику `/v1/messages`:

```python
tools = [
    {
        "name": "get_ga4_metric",
        "description": "Отримати метрику Google Analytics за вказаний період. Використовувати для питань про відвідуваність, перегляди, нових/повторних відвідувачів сайту nikvesti.com.",
        "input_schema": {
            "type": "object",
            "properties": {
                "metric": {
                    "type": "string",
                    "enum": ["activeUsers", "screenPageViews", "screenPageViewsPerSession", "newUsers", "returningUsers"]
                },
                "period": {
                    "type": "string",
                    "description": "Стандартний період АБО 'custom' якщо вказані start_date/end_date"
                },
                "start_date": {"type": "string", "description": "YYYY-MM-DD, тільки якщо period='custom'"},
                "end_date": {"type": "string", "description": "YYYY-MM-DD, тільки якщо period='custom'"}
            },
            "required": ["metric", "period"]
        }
    },
    # ... інші tools
]
```

Бот виконує цикл: викликати Claude → якщо відповідь містить `tool_use` блоки → виконати відповідні Python-функції → передати результати назад як `tool_result` → повторити, поки Claude не поверне фінальну текстову відповідь (без нових tool_use).

**Важливо:** обмежити кількість ітерацій цього циклу (наприклад, максимум 3-4), щоб уникнути нескінченних ланцюжків викликів при некоректному розумінні запиту.

## Тригер: коли повідомлення йде в Intent Router

Потрібно явно визначити (і це питання для обговорення з Олегом перед кодингом):

- Тільки повідомлення, що згадують бота (`@nikvesti_desk_bot` або reply на повідомлення бота)?
- Чи всі повідомлення в чаті редакції, які не є командами?
- Чи потрібен whitelist користувачів (тільки команда редакції, не випадкові учасники)?

Це впливає на UX і на ризик випадкових/неочікуваних викликів дорогого API.

## Ризики і що перевірити перед запуском

1. **Вартість** — кожен природномовний запит це мінімум 2 виклики Claude API (інтент + фінальна відповідь) плюс виклики GA4/Meta API. При активному використанні редакцією — порахувати орієнтовну вартість.
2. **Хибне розуміння періоду/метрики** — Claude може неправильно інтерпретувати розмовний період ("останнім часом", "цього тижня" коли тиждень ще не закінчився). Потрібне тестування на реальних формулюваннях питань від редакції (за принципом тестування парсерів на реальних даних — не вигадувати приклади).
3. **MySQL-пошук** — точність повнотекстового пошуку українською мовою (морфологія, відмінки — "Сєнкевича" vs "Сєнкевич") може бути слабкою з простим `LIKE`. Можливо знадобиться нормалізація запиту (стемінг/лематизація) або підключення зовнішнього пошукового рушія пізніше.
4. **Безпека БД-доступу** — пошук по сайту читає реальну продакшн-БД редакційної системи. Має бути read-only доступ, без жодної можливості запису.
5. **Розмежування з існуючими командами** — переконатися що Intent Router не перехоплює повідомлення, які мали б оброблятися існуючими handler'ами або conversation flow.
