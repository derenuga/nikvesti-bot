## МОДУЛЬ /STAT (статистика конкретного матеріалу) — детальний опис

**Належить до проекту:** Лис Микита — бот редакції МикВісті (див. PROJECT_CONTEXT.md).

### Що робить
За URL матеріалу nikvesti.com збирає і відображає статистику з трьох джерел:
- **Facebook** — ВСІ публікації про матеріал (звичайні пости + рілзи з посиланням в описі), кожна з переглядами/реакціями/коментарями/шерами; якщо їх кілька — нумерація і "Разом переглядів"
- **Telegram** — пост у каналі @nikvesti з посиланням на матеріал + його перегляди
- **GA4** — перегляди по мовних версіях (ua/ru/en) і загальна сума

### Файли коду
```
handlers/stat.py          — оркестрація: Facebook-пошук, GA4 запит, Telegram, форматування
handlers/telegram_stats.py — перегляди в Telegram: індекс постів каналу + пошук по t.me/s
handlers/storage.py        — ключ "tg_posts": {article_id: {"message_id": N}}
```

### Команди бота
- `/stat <url>` — наприклад: `/stat https://nikvesti.com/news/justice/320102-...`

### Env-змінні
- FACEBOOK_PAGE_TOKEN
- FACEBOOK_PAGE_ID = 301719373180657
- GA4_PROPERTY_ID = 321381722
- GA4_CREDENTIALS (JSON)

### Технічні деталі

**Витягування ID матеріалу:**
- З URL парситься числовий ID: `/news/justice/320102-...` → `"320102"`
- Regex: `r'/(\d{4,})-'`
- Використовується для пошуку в GA4 і для зіставлення з постом Facebook

**Telegram (перегляди поста в каналі):**
- Bot API принципово не дає ні пошуку по історії каналу, ні переглядів — обхід через веб-дзеркало публічних каналів
- Два шари:
  1. **Індекс**: `channel_post_handler` у bot.py отримує кожен пост каналу; якщо в ньому є посилання на nikvesti.com (текст, entities, кнопки) — зберігається `article_id → message_id` у storage (`tg_posts`, кап 2000 записів). Для матеріалів, запощених після деплою, пошук миттєвий
  2. **Пошук**: для старіших — гортаємо `https://t.me/s/nikvesti` назад (`?before=<msg_id>`, до ~300 постів), шукаємо href з `/{article_id}-`. Знайдене кладемо в індекс
- Перегляди: `span.tgme_widget_message_views` ("12.3K" → 12300); для індексного шляху — embed-сторінка `t.me/nikvesti/<id>?embed=1`
- ⚠️ Парсер написано за відомою розміткою t.me без живого тесту (мережа пісочниці не пускає до t.me) — якщо Telegram змінить розмітку, лагодити `_parse_message_block`/`_parse_views_text`

**Facebook:**
- API: Graph API v25.0
- Збирає ВСІ публікації, не тільки першу знайдену:
  - **Пости**: `GET /{page_id}/posts` за останні 14 днів; матч по message/story
  - **Рілзи**: `GET /{page_id}/video_reels` (останні ~50); матч по description;
    реакції/коменти/шери — `video_insights` (як у тижневому звіті),
    перегляди — метрика `blue_reels_play_count` з фолбеком `post_video_views`
- Матч точний: повний clean URL або `/{article_id}-` (щоб 320362 не ловив 1320362)
- Якщо публікацій кілька — нумеровані блоки (🎬 для рілзів) + "Разом переглядів"
- Метрики: `post_media_view` (перегляди, актуальна з листопада 2025), `post_reactions_by_type_total`, `post_comments`, `post_shares`
- Permalink формат: `facebook.com/nikvesti/posts/{id}`

**GA4:**
- Фільтр: `pagePath CONTAINS "/{article_id}-"` (без фільтру по Сінгапуру — для /stat показуємо реальні цифри)
- Метрика: `screenPageViews`
- Виміри: `pagePath` (для розрізнення мовних версій)
- Розрізнення мов: `/ua/`, `/ru/`, `/en/` в шляху
- Діапазон: за весь час (`2020-01-01` до `today`)

### Відомі обмеження і плани

- **TG post ID** — не зберігається в поточній реалізації. Після підключення БД сайту можна буде швидко знаходити Facebook-пост за TG ID статті (беклог п.11).
- **Facebook пошук по 14 днях** — старіші матеріали не знайдуться. Для архівної аналітики потрібна БД.
- **Facebook App Review** — після завершення буде доступна метрика `admin_creator` (автор поста).
