## МОДУЛЬ /STAT (статистика конкретного матеріалу) — детальний опис

**Належить до проекту:** Лис Микита — бот редакції МикВісті (див. PROJECT_CONTEXT.md).

### Що робить
За URL матеріалу nikvesti.com збирає і відображає статистику з чотирьох джерел:
- **Facebook** — ВСІ публікації про матеріал (звичайні пости + рілзи з посиланням в описі), кожна з переглядами/реакціями/коментарями/шерами; якщо їх кілька — нумерація і "Разом переглядів"
- **Instagram** — допис про матеріал, знайдений СЕМАНТИЧНО (по підпису, бо URL у стрічці немає) + перегляди/охоплення/лайки/коментарі/поширення/збереження
- **TikTok** — відео про матеріал, той самий семантичний пошук (TikTok = дзеркало інсти); перегляди/лайки/коментарі/поширення. Показується лише коли налаштовано TikTok OAuth
- **YouTube** — відео про матеріал, той самий семантичний пошук; перегляди/лайки/коментарі. Показується лише коли налаштовано YouTube OAuth
- **Telegram** — пост у каналі @nikvesti з посиланням на матеріал + його перегляди
- **GA4** — перегляди по мовних версіях (ua/ru/en) і загальна сума

### Файли коду
```
handlers/stat.py           — оркестрація: Facebook-пошук, Instagram, GA4 запит, Telegram, форматування
handlers/stat_instagram.py — семантичний пошук допису Instagram про матеріал (сигнатура ↔ підпис); ядро скорингу/сигнатури/судді переиспользує і stat_tiktok
handlers/stat_tiktok.py    — семантичний пошук відео TikTok (дзеркало інсти); те саме ядро, кандидати з TikTok Display API
handlers/stat_youtube.py   — семантичний пошук відео YouTube; те саме ядро, кандидати з YouTube Data API
handlers/instagram.py      — get_media_in_window (кандидати у вікні дат), get_media_insights (метрики допису)
handlers/tiktok_analytics.py — get_videos_in_window (відео у вікні дат: video_description+title+метрики)
handlers/youtube_analytics.py — get_videos_in_window (плейлист завантажень → videos.list: title+description+метрики)
handlers/telegram_stats.py — перегляди в Telegram: індекс постів каналу + пошук по t.me/s
handlers/storage.py        — ключ "tg_posts": {article_id: {"message_id": N}}
```

### Продуктивність (оптимізація 19.07.2026)

Було ~10 с на прогін: усі канали йшли ПОСЛІДОВНО (`await` за `await`), а сторінку
статті тягли до 4 разів (дата + сигнатура окремо в IG/TikTok/YouTube). Стало:
- **Один фетч сторінки** — `_fetch_article_context(url)` віддає і `pub_date`, і
  `signature` (заголовок+лід); `/stat` роздає сигнатуру в IG/TikTok/YouTube
  (параметр `sig`), тож вони більше не фетчать сторінку самі.
- **Усі канали паралельно** — `asyncio.gather` (FB, GA4, Telegram, Instagram,
  TikTok, YouTube незалежні): сумарна затримка = максимальна, а не сума.
  `return_exceptions=True` — збій одного каналу не валить решту (fallback на кожен).

Орієнтовно ~10 с → ~3–4 с (обмежено найповільнішим каналом, зазвичай Facebook —
пагінація стрічки постів + інсайти).

### Технічні деталі

**Витягування ID матеріалу:**
- З URL парситься числовий ID, два формати: зі слагом `/news/justice/320102-...` → `"320102"` (ID перед дефісом) і БЕЗ слага — старі матеріали, зокрема `/ru/…`: `/news/politics/111079` → `"111079"` (ID — останній сегмент шляху; URL спершу чиститься від query/anchor/трейлінг-слеша)
- Regex: `r'/(\d{4,})-'`, фолбек `r'/(\d{4,})$'` по clean URL
- Використовується для пошуку в GA4 і для зіставлення з постом Facebook

**Telegram (перегляди поста в каналі):**
- Bot API принципово не дає ні пошуку по історії каналу, ні переглядів — обхід через веб-дзеркало публічних каналів `t.me/s/nikvesti`
- `get_tg_stat(article_id, pub_date)` — три шляхи по черзі:
  1. **Індекс** (миттєво): `channel_post_handler` у bot.py індексує кожен новий пост каналу з посиланням на nikvesti.com (`article_id → message_id`, ключ `tg_posts`, кап 20000). Перегляди — з embed `t.me/nikvesti/<id>?embed=1`
  2. **Прицільний пошук по даті** (`search_channel_post_near_date`): якщо матеріалу немає в індексі, але відома дата публікації (JSON-LD з FB-гілки) — оцінюємо `message_id` **лінійною інтерполяцією між калібрувальними якорями** `CALIBRATION_ANCHORS` (дата 1-го числа → message_id, дані від Олега) і гортаємо вікно навколо оцінки (старт `оцінка + 300`, до 25 сторінок вниз). Так старий матеріал знаходиться за ~десяток запитів замість сотень сторінок
  3. **Пошук від найновіших** (`search_channel_post`): фолбек для свіжих без дати — гортаємо стрічку назад до ~300 постів
- Знайдене будь-яким шляхом кладемо в індекс (наступний `/stat` — миттєвий)
- **Бэкфіл** `/stat_backfill [місяців]` (тільки редакція): разово гортає історію `t.me/s` і індексує всі `article_id → message_id` одним записом (`bulk_save_tg_posts`), keep-min-id (оригінальний пост статті). Стоп по даті-cutoff або якірному floor-id. Пейс 0.35с/сторінка. Після прогону майже всі `/stat` — миттєві
- Перегляди: `span.tgme_widget_message_views` ("12.3K" → 12300); дата поста — `<time datetime=...>`
- ⚠️ Парсинг `t.me/s` перевірено на живому (перегляди приходять), але поведінка пагінації `?before=` і прицільний пошук з пісочниці не тестувались — перший реальний тест на проді

**Facebook:**
- API: Graph API v25.0
- **Вікно пошуку — від дати публікації матеріалу**, а не "14 днів назад": бот
  іде на сторінку статті, витягує `datePublished` з JSON-LD (фолбек —
  `<meta article:published_time>`) і шукає пости у вікні `[дата−1, дата+10 днів]`.
  Так знаходяться і старі матеріали. Якщо дату визначити не вдалось —
  відкат на дефолтне вікно 14 днів назад.
- Збирає ВСІ публікації про матеріал, не тільки першу знайдену:
  - **Пости**: `GET /{page_id}/posts` у вікні від дати публікації, **з пагінацією** (`paging.next`, до 15 сторінок × 100). Без гортання старий край вікна не потрапляв у вибірку: FB віддає найновіші перші, а на активній сторінці (~20 постів/день) вікно в 11 днів має 200+ постів — пост давньої статті висить на 2-3 сторінці
  - **Матч** точний: повний clean URL або `/{article_id}-` (щоб 320362 не ловив 1320362), перевіряється в `message`, `story` **і в URL вкладень** (`unshimmed_url`/`target.url` — посилання у ФБ часто не в тексті підпису, а прикріплене лінком; shimmed `l.facebook.com/l.php?u=...` декодується через `unquote`)
  - **Рілзи**: `GET /{page_id}/video_reels` (останні ~50); матч по description; реакції/коменти/шери — `video_insights` (як у тижневому звіті), перегляди — метрика `blue_reels_play_count` з фолбеком `post_video_views`
  - **Дедуплікація рілз↔пост**: FB дублює рілз ще й у стрічці постів (як відео-пост з іншим лічильником переглядів). Дубль прибираємо, лишаємо рілз (там коректний реловий лічильник). Зіставлення: `attachments.target.id` поста = ID рілза (+ страховка по числах у permalink). Рілз, який пагінація рілзів не дістала (напр. дату статті не визначено → лише найновіша сторінка) → відео-пост лишається як пост (фолбек)
  - Якщо публікацій кілька — нумеровані блоки (🎬 для рілзів) + "Разом переглядів"
- **Стійкість до збоїв Graph API**: `_get_fb_posts` має таймаут 30с і один ретрай на таймаут (сторінка з summary-полями інколи відповідає довго). Якщо стрічка постів усе одно впала — рілзи, що вже зібрані, лишаються, а помилка повертається окремо: у виводі «⚠️ Facebook відповідав надто довго / тимчасово обмежив запити… Спробуйте /stat ще раз за хвилину», а НЕ «поста немає»
- **Діагностика при порожньому FB** (щоб не гадати, коли справді немає поста): у виводі видно, чи визначилась дата й скільки постів переглянуто — `(шукав біля 31.05.2026, переглянув 217 постів у вікні + рілзи)`. Багато постів і нема збігу → проблема матчингу; мало → вікно/пагінація не дотяглись; «дату не визначив → 14 днів» → JSON-LD не розпарсився. Відсутність лічильника «переглянув N» = був виняток (див. окреме повідомлення про помилку)
- **Рілзи — те саме вікно, що пости (gap «останні ~50» закрито 19.07.2026):** `/video_reels` не має фільтра since/until і віддає від найновіших — тому гортаємо `paging.next` (до 15×100) з ранньою зупинкою, щойно дійшли до рілзів старших за вікно. Рілз знаходиться на будь-якій глибині. Виняток — дату статті не визначено: тоді лише найновіша сторінка (100), а рілз-дубль зі стрічки постів покажеться як пост (фолбек)
- Метрики: `post_media_view` (перегляди), `post_reactions_by_type_total`, `post_comments`, `post_shares`; permalink `facebook.com/nikvesti/posts/{id}`

**Instagram (семантичний пошук — `handlers/stat_instagram.py`):**
- **Проблема:** у стрічці Instagram НЕМАЄ посилання на статтю (діляться візуалом/контентом, а не URL) — матчити по URL, як у FB/TG, нічим.
- **Рішення:** зіставляємо ПО СМИСЛУ. Ключове спостереження на живих парах — підпис допису в інсті майже дослівно повторює лід статті, а лід лежить у `<meta name="description">` сторінки. Тобто сигнатура статті (`заголовок` з og:title + `лід` з meta description) ≈ підпис допису.
- **Двоступенева схема (дешева):**
  1. **Кандидати за датою** — `get_media_in_window(since, until)` тягне дописи інсти у вікні `[дата−1, дата+5]` (інста постить у той самий день, вікно вузьке → кілька-десяток кандидатів). Пагінація `paging.next`, до 6 сторінок × 100.
  2. **Лексична схожість** підпису до сигнатури: `max(coverage, bigram-Jaccard, reverse-coverage)`, де `coverage` = частка значущих слів статті (без стоп-слів, len≥3), наявних у підписі — головний сигнал для довгих підписів (інста переписує лід). **`reverse coverage`** (частка слів ПІДПИСУ, наявних у статті, з дисконтом `len/12` і порогом ≥5 токенів) — для КОРОТКИХ підписів: заголовок YouTube-шортса — витяжка із заголовка статті, пряма coverage структурно низька (7 слів не покриють 25-слівну сигнатуру; реальний кейс — шортс «Чому в зоопарку Миколаєва обидва слони чоловічої статі?» мав 0.29 і різався порогом, з reverse — 0.58 і авто-матч). Дисконт довжини тримає короткі генерик-підписи («новини Миколаєва сьогодні») поза ACCEPT: 5-6 повністю «статейних» слів → лише сіра зона (суддя), авто-збіг від ~7.
     - `score ≥ ACCEPT (0.50)` → впевнений збіг **без AI**. Кілька дописів близько до топа (пост+рілз однієї новини мають однаковий підпис) → показуємо всі (кап 3).
     - `JUDGE_MIN (0.28) ≤ score < ACCEPT` → **сіра зона**: топ-4 підписи віддаємо Claude-судді (Haiku, `fox_generate`) — «який допис про цю новину? число або 0». Стійко до сильного перепису / зміни мови.
     - `score < JUDGE_MIN` → допису немає.
- **Калібрування порогів** (на реальних парах D&G і Кім-міністр): вірні збіги `0.76`/`0.79`, будь-який чужий підпис того ж дня (погода/спорт/інша новина) `≤0.18` — величезний зазор, тому `ACCEPT` свідомо високий, а не «ловимо все». `SequenceMatcher` як сигнал відкинули — давав хибний шум (0.18 на погоді).
- **Метрики знайденого допису** — `get_media_insights(media_id)`: пробує набір метрик від багатого (`views,reach,likes,comments,shares,saved`) до мінімального (`reach`) — імена різняться за типом медіа й версією API. У виводі: 👁 Перегляди (views), 👀 Охоплення (reach), ❤️ Лайки, 💬 Коментарі, ✈️ Поширення (shares — «самолётик»), 🔖 Збереження (saved). shares/saved беруться ЛИШЕ з insights (у полях медіа їх немає); якщо API їх не віддав — рядок просто не показується. Перегляди входять у «Сукупно по всіх каналах».
- ⚠️ **Живі виклики Graph API (`/media` since/until, `/insights` метрики) на реальному API не тестувались** — перший реальний тест на проді (як свого часу TG-гілка). Деградація на випадок відсутніх метрик закладена.
- **Опційність:** без `INSTAGRAM_TOKEN` секція просто повертає порожньо (як FB без токена).

**TikTok (семантичний пошук — `handlers/stat_tiktok.py`):**
- **Той самий підхід, що Instagram.** TikTok у нас — дзеркало інсти: у відео той самий контент і той самий підпис, що в допису інсти, а він ≈ лід статті. URL статті у відео немає → зіставляємо по смислу.
- **Переиспользує ядро stat_instagram** — `_norm_tokens`, `_score` (max coverage/bigram-Jaccard), `get_article_signature`, `_judge` (з `platform="TikTok"`) і ті самі пороги/вікно. Тут лише інше джерело кандидатів і пакування метрик.
- **Кандидати** — `tiktok_analytics.get_videos_in_window(since, until)`: гортає Display API `/v2/video/list/` від найновіших, зупиняється на відео старших за `since`. Підпис для зіставлення — `video_description` + `title`.
- **Метрики** з того ж списку: 👁 Перегляди (view_count), ❤️ Лайки, 💬 Коментарі, ✈️ Поширення (share_count). Охоплення й збереження TikTok в API **не віддає** (лише в дашборді застосунку) — тому їх немає.
- **Опційність:** блок TikTok з'являється, лише коли налаштовано TikTok OAuth (`tiktok_analytics.is_configured()` — SANDBOX, `/tiktok_auth`). Без OAuth `get_tiktok_stat` повертає None і блок ховається (щоб не показувати «не знайдено» там, де мережа просто не під'єднана).
- ⚠️ Живі виклики Display API на пошук по вікну для /stat не тестувались окремо — перший реальний тест на проді (механіка списку та сама, що в місячному зрізі таблиці аналітики).

**YouTube (семантичний пошук — `handlers/stat_youtube.py`):**
- **Той самий підхід і ядро**, що Instagram/TikTok. URL статті у відео немає → зіставляємо сигнатуру статті з `title + description` відео.
- **Кандидати** — `youtube_analytics.get_videos_in_window(since, until)`: бере плейлист завантажень каналу (channels.list `mine=true` → uploads), гортає `playlistItems` від найновіших до межі `since`, потім добирає `snippet+statistics` пачками по 50 (`videos.list`). Токен — той самий OAuth `yt-analytics`, що для таблиці аналітики (Data API його приймає для `mine=true`, як у get_channel_stats).
- **Метрики**: 👁 Перегляди (viewCount), ❤️ Лайки (likeCount), 💬 Коментарі (commentCount). Поширень і охоплення на рівні відео Data API не дає.
- **Опційність:** блок YouTube — лише коли налаштовано YouTube OAuth (`youtube_analytics.is_configured()`); інакше `get_youtube_stat` повертає None і блок ховається. Помилка листингу API (напр. 403 по scope) → теж None (ховаємо блок), а НЕ «Відео не знайдено» — помилка API ≠ відсутність відео (так само в TikTok).
- **Застереження:** опис відео на YT часто містить сталий текст (лінки на соцмережі, підписка) — але coverage рахує частку слів СТАТТІ в описі, тож сталий текст збіг не роздуває. ⚠️ Живі виклики Data API на пошук по вікну для /stat не тестувались — перший реальний тест на проді; якщо scope токена не покриє Data API-лістинг (можливий 403) — треба дозволити `youtube.readonly` при перевипуску refresh token.

**GA4:**
- Фільтр: `pagePath CONTAINS "/{article_id}-"` (без фільтру по Сінгапуру — для /stat показуємо реальні цифри)
- Метрика: `screenPageViews`
- Виміри: `pagePath` (для розрізнення мовних версій)
- Розрізнення мов: `/ua/`, `/ru/`, `/en/` в шляху
- Діапазон: за весь час (`2020-01-01` до `today`)

### Відомі обмеження і плани

- **TG post ID** — не зберігається в поточній реалізації. Після підключення БД сайту можна буде швидко знаходити Facebook-пост за TG ID статті (беклог п.11).
- **Facebook пошук по 14 днях** — старіші матеріали не знайдуться. Для архівної аналітики потрібна БД.
- **Facebook App Review** — після завершення буде доступна метрика `admin_creator` (автор поста).
- **Instagram — рілз поза вікном/без підпису:** якщо допис узагалі без тексту (лише візуал) або підпис радикально відрізняється від ліда — лексика не спрацює, суддя теж може не впізнати. Візуальний фолбек (og:image статті ↔ картинка допису через Claude vision) — можливий другий сигнал на майбутнє, але інста часто переверстує візуал, тож це тай-брейкер, не основа.

### Снімок у Нору + швидкий шлях (реалізовано 19.07.2026)

Таблиця **`article_stats`** у Норі (bot_db): PK `(article_id, channel, object_id)`, `item` JSONB (весь елемент виводу як є: permalink, дата, метрики, method), `captured_at`. Канали: `site` (by_lang GA4), `facebook`/`instagram`/`tiktok`/`youtube` (по рядку на знайдений пост/рілз/відео, object_id = id у платформи), `telegram` (url+views). Логіка — `handlers/stat_store.py`:

- **Upsert «останній стан», без історії** (рішення Олега): кожен `/stat` перезаписує снімок каналу (`replace_channel_stats`: DELETE+INSERT). Порожній свіжий результат канал НЕ затирає — збій пошуку/матчингу не стирає робочий індекс. Фолбек-дані з Нори (позначені `nora`) не пересохраняються.
- **Швидкий шлях**: наступний `/stat` читає індекс (`load_index`) і для каналів із збереженими object_id минає ВЕСЬ пошук (вікна дат, листинги, семантичний скоринг, суддю) — метрики тягнуться одразу по id: FB `get_fb_stats_by_objects` (пост: GET /{id}?fields=… + insights; рілз: video_insights), IG `get_instagram_stat_by_ids` (`get_media_item` + insights), TikTok `get_tiktok_stat_by_ids` (`/v2/video/query/` filters.video_ids), YouTube `get_youtube_stat_by_ids` (videos.list по id). Коли всі соцканали в індексі — сторінка статті взагалі не фетчиться. Перший `/stat` ~8-9с, повторні ~2-3с. `method: "index"` у діагностиці.
- **Фолбек «з Нори»**: збій живого каналу (виняток, ліміт FB) при наявному снімку → показуємо снімок з поміткою «🗄 живе джерело не відповіло — знімок з Нори від DD.MM HH:MM» замість «не знайдено»/помилки. GA4-збій → site з снімка (без помітки в поточній версії — цифри по мовах ті самі).
- Канал, якого немає в індексі (не знайшли минулого разу), шукається повним шляхом щоразу — пост міг з'явитися пізніше.
- Без `BOT_DATABASE_URL` все тихо вимкнено, `/stat` працює як раніше.
- Уточнення по GA4: запит `/stat` (`pagePath` + `screenPageViews`) GA4 тримає фактично безстроково — data-retention (2–14 міс) ріже лише event-level дані для Explorations. Снімок цінний як фіксація/швидкість/страховка, а не єдиний рятунок.

### Беклог: швидкість /stat (решта ідей)

1. **Внутрішньоканальні ланцюжки розпаралелити**: у FB — рілзи і стрічка постів послідовні, можна gather; у YouTube — чанки `videos.list` можна gather.
2. **Метрики знайдених дописів паралельно**: insights по одному на об'єкт послідовно — при 2-3 збігах можна gather.
3. **Обрізати вікна листингів**: YouTube `max_pages=6×50` надлишково для вікна в 6 днів; TikTok `25×20` — так само.
4. **Прогрес-повідомлення** (UX): «⏳ …ФБ ✓, інста ✓, ютуб…» — прибирає відчуття «висить».
