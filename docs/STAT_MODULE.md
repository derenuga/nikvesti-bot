## МОДУЛЬ /STAT (статистика конкретного матеріалу) — детальний опис

**Належить до проекту:** Лис Микита — бот редакції МикВісті (див. PROJECT_CONTEXT.md).

### Що робить
За URL матеріалу nikvesti.com збирає і відображає статистику з чотирьох джерел:
- **Facebook** — ВСІ публікації про матеріал (звичайні пости + рілзи з посиланням в описі), кожна з переглядами/реакціями/коментарями/шерами; якщо їх кілька — нумерація і "Разом переглядів"
- **Instagram** — допис про матеріал, знайдений СЕМАНТИЧНО (по підпису, бо URL у стрічці немає) + перегляди/охоплення/лайки/коментарі
- **Telegram** — пост у каналі @nikvesti з посиланням на матеріал + його перегляди
- **GA4** — перегляди по мовних версіях (ua/ru/en) і загальна сума

### Файли коду
```
handlers/stat.py           — оркестрація: Facebook-пошук, Instagram, GA4 запит, Telegram, форматування
handlers/stat_instagram.py — семантичний пошук допису Instagram про матеріал (сигнатура ↔ підпис)
handlers/instagram.py      — get_media_in_window (кандидати у вікні дат), get_media_insights (метрики допису)
handlers/telegram_stats.py — перегляди в Telegram: індекс постів каналу + пошук по t.me/s
handlers/storage.py        — ключ "tg_posts": {article_id: {"message_id": N}}
```

### Команди бота
- `/stat <url>` — наприклад: `/stat https://nikvesti.com/news/justice/320102-...`

### Env-змінні
- FACEBOOK_PAGE_TOKEN
- FACEBOOK_PAGE_ID = 301719373180657
- GA4_PROPERTY_ID = 321381722
- GA4_CREDENTIALS (JSON)

### Технічні деталі

**Витягування ID матеріалу:**
- З URL парситься числовий ID: `/news/justice/320102-...` → `"320102"`
- Regex: `r'/(\d{4,})-'`
- Використовується для пошуку в GA4 і для зіставлення з постом Facebook

**Telegram (перегляди поста в каналі):**
- Bot API принципово не дає ні пошуку по історії каналу, ні переглядів — обхід через веб-дзеркало публічних каналів `t.me/s/nikvesti`
- `get_tg_stat(article_id, pub_date)` — три шляхи по черзі:
  1. **Індекс** (миттєво): `channel_post_handler` у bot.py індексує кожен новий пост каналу з посиланням на nikvesti.com (`article_id → message_id`, ключ `tg_posts`, кап 20000). Перегляди — з embed `t.me/nikvesti/<id>?embed=1`
  2. **Прицільний пошук по даті** (`search_channel_post_near_date`): якщо матеріалу немає в індексі, але відома дата публікації (JSON-LD з FB-гілки) — оцінюємо `message_id` **лінійною інтерполяцією між калібрувальними якорями** `CALIBRATION_ANCHORS` (дата 1-го числа → message_id, дані від Олега) і гортаємо вікно навколо оцінки (старт `оцінка + 300`, до 25 сторінок вниз). Так старий матеріал знаходиться за ~десяток запитів замість сотень сторінок
  3. **Пошук від найновіших** (`search_channel_post`): фолбек для свіжих без дати — гортаємо стрічку назад до ~300 постів
- Знайдене будь-яким шляхом кладемо в індекс (наступний `/stat` — миттєвий)
- **Бэкфіл** `/stat_backfill [місяців]` (тільки редакція): разово гортає історію `t.me/s` і індексує всі `article_id → message_id` одним записом (`bulk_save_tg_posts`), keep-min-id (оригінальний пост статті). Стоп по даті-cutoff або якірному floor-id. Пейс 0.35с/сторінка. Після прогону майже всі `/stat` — миттєві
- Перегляди: `span.tgme_widget_message_views` ("12.3K" → 12300); дата поста — `<time datetime=...>`
- ⚠️ Парсинг `t.me/s` перевірено на живому (перегляди приходять), але поведінка пагінації `?before=` і прицільний пошук з пісочниці не тестувались — перший реальний тест на проді

**Facebook:**
- API: Graph API v25.0
- **Вікно пошуку — від дати публікації матеріалу**, а не "14 днів назад": бот
  іде на сторінку статті, витягує `datePublished` з JSON-LD (фолбек —
  `<meta article:published_time>`) і шукає пости у вікні `[дата−1, дата+10 днів]`.
  Так знаходяться і старі матеріали. Якщо дату визначити не вдалось —
  відкат на дефолтне вікно 14 днів назад.
- Збирає ВСІ публікації про матеріал, не тільки першу знайдену:
  - **Пости**: `GET /{page_id}/posts` у вікні від дати публікації, **з пагінацією** (`paging.next`, до 15 сторінок × 100). Без гортання старий край вікна не потрапляв у вибірку: FB віддає найновіші перші, а на активній сторінці (~20 постів/день) вікно в 11 днів має 200+ постів — пост давньої статті висить на 2-3 сторінці
  - **Матч** точний: повний clean URL або `/{article_id}-` (щоб 320362 не ловив 1320362), перевіряється в `message`, `story` **і в URL вкладень** (`unshimmed_url`/`target.url` — посилання у ФБ часто не в тексті підпису, а прикріплене лінком; shimmed `l.facebook.com/l.php?u=...` декодується через `unquote`)
  - **Рілзи**: `GET /{page_id}/video_reels` (останні ~50); матч по description; реакції/коменти/шери — `video_insights` (як у тижневому звіті), перегляди — метрика `blue_reels_play_count` з фолбеком `post_video_views`
  - **Дедуплікація рілз↔пост**: FB дублює рілз ще й у стрічці постів (як відео-пост з іншим лічильником переглядів). Дубль прибираємо, лишаємо рілз (там коректний реловий лічильник). Зіставлення: `attachments.target.id` поста = ID рілза (+ страховка по числах у permalink). Рілз поза останніми ~50 (не в /video_reels) → відео-пост лишається як пост (фолбек)
  - Якщо публікацій кілька — нумеровані блоки (🎬 для рілзів) + "Разом переглядів"
- **Стійкість до збоїв Graph API**: `_get_fb_posts` має таймаут 30с і один ретрай на таймаут (сторінка з summary-полями інколи відповідає довго). Якщо стрічка постів усе одно впала — рілзи, що вже зібрані, лишаються, а помилка повертається окремо: у виводі «⚠️ Facebook відповідав надто довго / тимчасово обмежив запити… Спробуйте /stat ще раз за хвилину», а НЕ «поста немає»
- **Діагностика при порожньому FB** (щоб не гадати, коли справді немає поста): у виводі видно, чи визначилась дата й скільки постів переглянуто — `(шукав біля 31.05.2026, переглянув 217 постів у вікні + рілзи)`. Багато постів і нема збігу → проблема матчингу; мало → вікно/пагінація не дотяглись; «дату не визначив → 14 днів» → JSON-LD не розпарсився. Відсутність лічильника «переглянув N» = був виняток (див. окреме повідомлення про помилку)
- **Відомий gap:** рілзи скануються лише в останніх ~50 (у `/video_reels` немає надійного фільтра since/until) — для старих матеріалів рілз може не знайтись; його відео-версія з ленти постів усе одно покажеться як пост
- Метрики: `post_media_view` (перегляди), `post_reactions_by_type_total`, `post_comments`, `post_shares`; permalink `facebook.com/nikvesti/posts/{id}`

**Instagram (семантичний пошук — `handlers/stat_instagram.py`):**
- **Проблема:** у стрічці Instagram НЕМАЄ посилання на статтю (діляться візуалом/контентом, а не URL) — матчити по URL, як у FB/TG, нічим.
- **Рішення:** зіставляємо ПО СМИСЛУ. Ключове спостереження на живих парах — підпис допису в інсті майже дослівно повторює лід статті, а лід лежить у `<meta name="description">` сторінки. Тобто сигнатура статті (`заголовок` з og:title + `лід` з meta description) ≈ підпис допису.
- **Двоступенева схема (дешева):**
  1. **Кандидати за датою** — `get_media_in_window(since, until)` тягне дописи інсти у вікні `[дата−1, дата+5]` (інста постить у той самий день, вікно вузьке → кілька-десяток кандидатів). Пагінація `paging.next`, до 6 сторінок × 100.
  2. **Лексична схожість** підпису до сигнатури: `max(coverage, bigram-Jaccard)`, де `coverage` = частка значущих слів статті (без стоп-слів, len≥3), наявних у підписі — головний сигнал, бо інста переписує лід.
     - `score ≥ ACCEPT (0.50)` → впевнений збіг **без AI**. Кілька дописів близько до топа (пост+рілз однієї новини мають однаковий підпис) → показуємо всі (кап 3).
     - `JUDGE_MIN (0.28) ≤ score < ACCEPT` → **сіра зона**: топ-4 підписи віддаємо Claude-судді (Haiku, `fox_generate`) — «який допис про цю новину? число або 0». Стійко до сильного перепису / зміни мови.
     - `score < JUDGE_MIN` → допису немає.
- **Калібрування порогів** (на реальних парах D&G і Кім-міністр): вірні збіги `0.76`/`0.79`, будь-який чужий підпис того ж дня (погода/спорт/інша новина) `≤0.18` — величезний зазор, тому `ACCEPT` свідомо високий, а не «ловимо все». `SequenceMatcher` як сигнал відкинули — давав хибний шум (0.18 на погоді).
- **Метрики знайденого допису** — `get_media_insights(media_id)`: пробує набір метрик від багатого (`views,reach,likes,comments,shares,saved`) до мінімального (`reach`) — імена різняться за типом медіа й версією API. У виводі: 👁 Перегляди (views), 👀 Охоплення (reach), ❤️ Лайки, 💬 Коментарі, ✈️ Поширення (shares — «самолётик»), 🔖 Збереження (saved). shares/saved беруться ЛИШЕ з insights (у полях медіа їх немає); якщо API їх не віддав — рядок просто не показується. Перегляди входять у «Сукупно по всіх каналах».
- ⚠️ **Живі виклики Graph API (`/media` since/until, `/insights` метрики) на реальному API не тестувались** — перший реальний тест на проді (як свого часу TG-гілка). Деградація на випадок відсутніх метрик закладена.
- **Опційність:** без `INSTAGRAM_TOKEN` секція просто повертає порожньо (як FB без токена).

**GA4:**
- Фільтр: `pagePath CONTAINS "/{article_id}-"` (без фільтру по Сінгапуру — для /stat показуємо реальні цифри)

- Виміри: `pagePath` (для розрізнення мовних версій)
- Розрізнення мов: `/ua/`, `/ru/`, `/en/` в шляху
- Діапазон: за весь час (`2020-01-01` до `today`)

### Відомі обмеження і плани

- **TG post ID** — не зберігається в поточній реалізації. Після підключення БД сайту можна буде швидко знаходити Facebook-пост за TG ID статті (беклог п.11).
- **Facebook пошук по 14 днях** — старіші матеріали не знайдуться. Для архівної аналітики потрібна БД.
- **Facebook App Review** — після завершення буде доступна метрика `admin_creator` (автор поста).
- **Instagram — рілз поза вікном/без підпису:** якщо допис узагалі без тексту (лише візуал) або підпис радикально відрізняється від ліда — лексика не спрацює, суддя теж може не впізнати. Візуальний фолбек (og:image статті ↔ картинка допису через Claude vision) — можливий другий сигнал на майбутнє, але інста часто переверстує візуал, тож це тай-брейкер, не основа.

### Беклог: снімок метрик у Нору (узгоджено з Олегом, 19.07.2026)

Зараз `/stat` **не** пише метрики в Нору і **не** читає їх звідти — GA4/FB/TG/IG тягнуться вживу щоразу, а в Норі осідає лише індекс `tg_posts` (article_id → message_id). Домовлені пункти:

1. **Снімок кожного `/stat` у Нору — upsert (останній стан, без історії).** Нова таблиця `article_stats`, ключ `(article_id, channel)` — по одному актуальному рядку на матеріал+канал (`site`/`facebook`/`instagram`/`telegram`): views, reach, reactions, comments, shares, permalink, raw JSONB, captured_at. Перезапис при кожному новому `/stat`. Дає: фіксацію цифри на момент запиту + джерело для фолбека.
2. **Фолбек-читання з Нори**, коли живе джерело віддало порожньо/помилку (напр. FB-ліміт, або GA4 з часом підчистить деталізацію по сторінці): показати останній відомий снімок із поміткою «дані з Нори від DD.MM».
   - Уточнення по GA4: запит `/stat` (`pagePath` + `screenPageViews`, стандартні виміри) GA4 тримає фактично безстроково — data-retention (2–14 міс) ріже лише event-level дані для Explorations, не агреговані стандартні звіти. Тобто нагальної загрози «GA4 почистить» для `/stat` немає; снімок у Нору цінний як фіксація/швидкість/страховка, а не як єдиний рятунок.
