# Long-term vision — ідеї на майбутнє

**Статус усіх пунктів нижче:** ідея, не в розробці. Це не тактичний беклог (для дрібних фіксів і наступних кроків — `PROJECT_CONTEXT.md`), а напрямки розвитку продукту й інфраструктури, що змінюють архітектуру або модель бота. Нічого з цього не кодити без явного "кодь" від Олега, навіть якщо пункт виглядає простим.

**Навіщо окремий файл:** ці ідеї за маштабом і ризиком відрізняються від "додати джерело конкурентів" чи "виправити текст ОВА" — вони передбачають нову інфраструктуру, гроші користувачів, або зміну характеру бота з внутрішнього інструменту редакції на продукт для зовнішніх користувачів. Змішувати їх з операційним беклогом в `PROJECT_CONTEXT.md` ускладнило б пріоритезацію щоденної роботи.

## Кластер 1: Монетизація — підписка для зовнішніх користувачів

Зараз Лис Микита — внутрішній інструмент редакції (`ALLOWED_USER_IDS`). Ідея: відкрити частину функціоналу зовнішнім підписникам як платний продукт.

### 1.1 Підписка на оновлення проєктів рішень, документів, оголошень, тендерів

Зараз моніторинг (`prozorro.py`, `documents.py`, `law_enforcement.py`, `competitors.py`) шле все в один канал `🦊 Микита винюхав` для редакції. Підписка означала б:
- Окремий механізм доставки для зовнішніх підписників (не той самий канал — або фільтрований feed, або персональні розсилки за інтересами: "тільки тендери", "тільки міськрада" тощо)
- Модель доступу: Telegram-бот з персональним whitelist по підписці, а не статичний `ALLOWED_USER_IDS`
- **Залежність:** потребує переходу від поточної моделі "один статичний список ID в env" до динамічного сховища підписників (див. кластер 2 — MySQL)

### 1.2 Промокоди для підписки

- Потрібна модель: код → термін дії / знижка / тип підписки
- Прив'язка коду до Telegram user ID при активації, облік використаних/невикористаних кодів
- **Залежність:** напряму залежить від 1.1 (немає підписки — нема що активувати промокодом) і від MySQL/БД для зберігання кодів і статусів (file storage на Railway Volume для цього не підходить — конкурентний доступ, потреба в транзакціях)

### Відкриті питання (обговорити з Олегом перед будь-яким кодингом)
- Платіжний шлюз — Telegram Stars, Stripe, локальний український еквайринг?
- Юридична форма (ФОП, оплата послуг) — поза межами технічної реалізації, але блокує запуск
- Які саме потоки даних варті платної підписки, а які лишаються безкоштовними публічними (репутаційна цінність відкритого каналу)

## Кластер 2: Інфраструктура — перехід з Railway+JSON на власний сервер+MySQL

Два пов'язані пункти, по суті один архітектурний крок:

### 2.1 Перехід на власний сервер

Зараз: Railway (проект `remarkable-stillness`), Railway Volume для стану. Обмеження Railway, які вже зафіксовані в операційному беклозі — відсутність статичного IP для whitelist зовнішньої БД (`PROJECT_CONTEXT.md`, п. 17 — "Перенести бота на KEY4"). Власний сервер (або KEY4, де вже хоститься сайт) вирішує:
- Статичний IP → прямий доступ до production БД сайту без REST-wrapper/SSH-тунелю
- Контроль над ресурсами для важчих задач (повнотекстовий пошук по 17 роках архіву — кластер 3)

### 2.2 Перехід з file storage на MySQL

Зараз: єдиний JSON-файл `/data/prozorro_state.json` на Railway Volume для стану всіх модулів моніторингу (`storage.py`). Працює для поточного масштабу (один файл, один процес), але:
- Не масштабується на підписників/промокоди (кластер 1) — потрібні транзакції, конкурентний запис, індекси
- Немає історії/аудиту — JSON зберігає тільки поточний стан "бачені ID", не лог подій
- MySQL вже є на сервері сайту (Laminas/Zend, KEY4) — логічно використати ту саму БД або сусідню на тому ж сервері, а не піднімати окрему

**Послідовність:** 2.1 (сервер) звичайно йде перед або разом з 2.2 (БД), оскільки власний сервер — передумова для надійного MySQL-з'єднання без проблем з мережею/IP, які зараз блокують прямий доступ до БД сайту.

### 2.3 Альтернатива міграції: доступ до MySQL сайту прямо з Railway

Обговорено 01.07.2026. Переїзд з Railway — не єдиний шлях до БД сайту; є способи
дати боту доступ, залишившись на Railway (що знімає головну причину міграції):

1. **Railway Static Outbound IP** (Pro-план): вмикається в налаштуваннях сервісу,
   дає постійний вихідний IP — саме те, що просять адміни для whitelist. Найпростіший
   шлях, перевірити доступність фічі на поточному плані.
2. **SSH-тунель**: бот сам відкриває вихідне SSH-з'єднання до сервера сайту
   (авторизація ключем, а не IP) і ходить у MySQL через тунель як через localhost.
   Обмежений SSH-користувач без shell, тільки port-forward на 127.0.0.1:3306.
   Python: бібліотека `sshtunnel`. Працює з будь-якого динамічного IP.
3. **REST-обгортка на сервері сайту**: маленький read-only API поруч із сайтом
   (авторизація токеном). Найбільше роботи, але БД взагалі не виходить назовні;
   стратегічно корисно якщо колись будуть підписки (кластер 1).
4. **Tailscale/WireGuard** — VPN-mesh між Railway-контейнером і сервером; робочий,
   але найбільше рухомих частин.

У будь-якому варіанті: окремий MySQL-користувач для бота, тільки SELECT на потрібні
таблиці, ліміти на запити (це production-БД сайту).

### 2.4 Що розблокується з доступом до БД (чеклист, станом на 02.07.2026)

Олег дав таску KEY4 на IP-доступ до БД сайту. Коли з'явиться — беремо ці пункти
(усі відкладені з ревізії саме через це, `REVIEW_2026_07.md`):

**Статус (03.07.2026): доступ отримано.** Окремий read-only користувач `nikvesti_bot`
(SELECT only на `nikvesti.*`, SSL обов'язковий, whitelist за IP Railway, ліміти на
запити). **Крок A зроблено** — тонкий адаптер `handlers/db.py` (PyMySQL, з'єднання на
запит, guard «тільки читання») + `/dbtest` для перевірки з'єднання. Далі — пункти нижче
поверх адаптера (спершу в.4 атрибуція авторів і а.7 контроль заголовків як найдешевші).

- **в.4 — атрибуція авторів у NLQ.** "Як зайшли статті Аліси за тиждень?" —
  автор береться з БД напряму, а не парсингом `<meta author>` з кожного URL
  (як зараз у `get_author_from_url`). Заодно прибрати повільний HTTP-парсинг автора.
- **а.7 — контроль якості заголовків.** Свіжі статті брати з БД (а не RSS/sitemap),
  Haiku перевіряє заголовок/лід на одруківки й росіянізми → тихий пост у чат.
- **а.6 / кластер 3 — пошук по архіву.** `search_site_articles` одразу поверх БД
  (17 років), а не публічний пошук/sitemap. Спершу LIKE/FULLTEXT; за потреби —
  вектори (див. кластер 3).
- **а.5 — "Тижневик Лиса".** Головна цінність — робота на сайті тиждень-до-тижня
  (кількість матеріалів, автори, теми) з БД; GA4/соцмережі/тендери — доповнення.
  Без БД тижневик неповний, тому чекає.
- **/stat і `get_ga4_top_articles`** — автор і метадані матеріалу з БД замість
  HTTP-парсингу; точніше й швидше.

## Кластер 3: AI-шар над 17-річним архівом новин МикВісті

Найамбітніший і найцінніший напрямок — перетворити Лиса з аналітика метрик (GA4/Search Console, кластер з `NATURAL_LANGUAGE_QUERIES_MODULE.md`) на експерта з контентом редакції.

### 3.1 Доступ до БД сайту з повною історією новин

- Read-only доступ до production БД МикВісті (Laminas/Zend, сервер KEY4) — сотні тисяч статей за 17 років
- Це той самий пункт, що вже в операційному беклозі (`PROJECT_CONTEXT.md`, п. 11 "БД сайту"), але контекст ширший ніж просто `/stat` — тут це фундамент для 3.2
- **Залежність:** прямо залежить від кластеру 2 (власний сервер / статичний IP) — без нього доступ можливий тільки через REST-wrapper чи SSH-тунель з усіма їх обмеженнями

### 3.2 Природномовні відповіді на основі повного архіву

Розширення Intent Router (`handlers/query_router.py`, описаний у `NATURAL_LANGUAGE_QUERIES_MODULE.md`) новим класом tools поверх БД сайту:
- `search_site_articles(query, ...)` — вже у беклозі `NATURAL_LANGUAGE_QUERIES_MODULE.md` як план, але в контексті 17-річного архіву це не "знайти останню статтю про X", а повноцінний RAG-шар: "Що ми писали про забудову Намиву за останні 10 років?", "Як змінювалась тема комунальних тарифів у наших матеріалах?"
- На обсязі сотень тисяч статей простий `LIKE`/`FULLTEXT` (як зафіксовано в плані NLQ-модуля) може не вистачити для якісного пошуку — ймовірно знадобиться:
  - Векторний пошук (embeddings) для семантичних запитів, а не тільки збіг ключових слів
  - Це новий шар інфраструктури (vector store) поверх MySQL — окреме архітектурне рішення, не частина початкового плану NLQ-модуля
- Для редакції це насамперед інструмент журналістської роботи (контекст для нових матеріалів, перевірка "чи ми це вже писали"), а не просто аналітика

### Зв'язок з іншими документами
- `docs/NATURAL_LANGUAGE_QUERIES_MODULE.md` — поточний контур (GA4 + Search Console) і вже задокументований план для `search_site_articles`/`get_article_by_url` — цей розділ (3.2) їх не дублює, а описує що змінюється в підході, коли масштаб архіву — 17 років, а не "останні кілька статей"
- `docs/PROJECT_CONTEXT.md`, п. 11, 12, 17 — операційні кроки, які є передумовою для цього кластеру

## Як ці кластери співвідносяться

```
Кластер 2 (сервер + MySQL)
        │
        ├──▶ Кластер 1 (підписка + промокоди) — потребує БД для облікових записів/кодів
        │
        └──▶ Кластер 3 (доступ до архіву + RAG-пошук) — потребує статичного IP і БД-доступу
```

Кластер 2 — спільна інфраструктурна передумова для обох інших. Має сенс розглядати його першим, якщо приймається рішення рухатись у цьому напрямку взагалі.
