# Mini App «Команда» — таски і KPI редакції

Telegram Mini App для контролю завдань і KPI команди (запит Олега, 24.07.2026).
Два інтерфейси: **менеджер** (Олег + головред — ставить таски, приймає роботу,
конструюватиме KPI) і **журналістка** (виконує таски, бачить свій прогрес).
Ключові рішення Олега:

- таски **персональні**, групуються по проєктах (вільний ярлик);
- KPI — **по відділах** (стрічка ≠ ліди ≠ соцмережі), хвиля 2;
- геймифікація: дизайн/іконки/конфеті — виконання має радувати;
- персональна аналітика «мої найкращі матеріали» — щоб журналістка бачила
  ефект своєї роботи (хвиля 3, дані вже є в daily_stats/article_stats).

## Архітектура

Один процес на Railway — два входи: python-telegram-bot (polling) + aiohttp
(HTTP) у спільному event loop. Сервер стартує з `post_init` і **тільки коли є
PORT** — без налаштованого домену модуль спить, деплой нічого не змінює.

```
handlers/team_roster.py   — ростер: відділи, менеджери, резолв initData → людина,
                            кеш tg_id (team_users) для пінгів
handlers/team_tasks.py    — таблиці team_tasks/team_task_events у Норі, CRUD,
                            життєвий цикл, пінги від Лиса (best-effort)
handlers/webapp.py        — aiohttp: перевірка підпису initData, API з правами,
                            статика, /health, команда /team
webapp/                   — SPA без збірки: index.html + style.css + app.js
```

### Життєвий цикл таска

`todo → doing → review → done` (+ `dropped`). Виконавиця рухає власний таск
`todo↔doing→review` і чіпляє лінк на матеріал; менеджер — усе (створення,
редагування, `done`/`dropped`, повернення з review). Права перевіряє API-шар.

### Авторизація

`Authorization: tma <initData>` у кожному запиті → перевірка HMAC-підпису
секретом від BOT_TOKEN (алгоритм Telegram) + свіжість `auth_date` ≤ 24 год →
резолв людини по ростеру (кеш team_users → відомі id → username). Не з
команди — 403 навіть із валідним підписом. Закритість чату, звідки відкрили
апку, захистом НЕ є — лінк t.me/бот/апка відкривається будь-де.

### Ідентифікація людей

У TEAM — лише @хендли; numeric id знаємо заздалегідь тільки для Олега, Каті,
Лізи, Марії, Сергія. Решта докешовується у `team_users` при **першому вході в
апку** (initData дає і id, і username). Наслідок: пінг журналістці працює після
того, як вона хоч раз відкрила апку І стартувала бота — до того пінги тихо
пропускаються (створення таска ніколи не падає через це; апка — джерело правди).

### Пінги від Лиса

- нове завдання → виконавиці (з 🔥 якщо терміново, з дедлайном);
- здано на перевірку → автору таска;
- прийнято → виконавиці «🎉 Микита пишається»;
- повернуто з перевірки → виконавиці.

## Налаштування (разово)

1. Railway → сервіс бота → Settings → Networking → **Generate Domain** —
   з'явиться `PORT` в env і публічний https-домен.
2. Env: `WEBAPP_URL = https://<домен>` (для кнопок «Відкрити»).
3. BotFather → `/newapp` для @mykvisti_bot → Web App URL = WEBAPP_URL,
   short name напр. `team` → env `WEBAPP_DIRECT_LINK = https://t.me/mykvisti_bot/team`
   (для запуску апки кнопкою з групових чатів).
4. Опційно BotFather → Bot Settings → Menu Button → той самий URL.
5. Бонус: `GET /health` віддає 200 — можна вказати як `VIBER_WEBHOOK_URL`
   і прибрати зовнішню залежність дзеркала Viber.

Перевірка: `/team` у приваті з ботом → кнопка → апка відкрилась, у шапці твоє
ім'я і відділ. Менеджерам видно таби і кнопку ＋.

## API

| Метод | Шлях | Хто | Що |
|---|---|---|---|
| GET | /api/me | всі | ім'я, відділ, роль, лічильники тижня |
| GET | /api/roster | всі | ростер для форм |
| GET | /api/tasks | всі | менеджер — усі; журналістка — свої (+done 30 дн) |
| POST | /api/tasks | менеджер | створити (пінг виконавиці) |
| PATCH | /api/tasks/{id} | за роллю | статус/поля; пінги по переходах |
| GET | /api/tasks/{id}/events | причетні | журнал змін |
| GET | /health | — | 200 ok (health-check, Viber webhook) |

## Схема (Нора, Postgres)

- `team_users(tg_id PK, username, person, first_seen, last_seen)` — кеш id;
- `team_tasks(id, project, title, body, assignee, creator, priority 0/1/2,
  status, deadline, article_url, created_at, updated_at, done_at)` — люди
  зберігаються канонічним ім'ям ростера (id до першого входу невідомі);
- `team_task_events(id, task_id, actor, event, detail, at)` — журнал
  (історія картки + паливо геймифікації).

Схема створюється ліниво з `team_tasks.ensure_team_schema()` — самодостатньо,
НЕ в `bot_db._SCHEMA_STATEMENTS` (модуль опційний, як budget_*).

## Дорожня карта

- **Хвиля 1 (зроблено):** каркас, auth, таски, два інтерфейси, пінги, /team.
- **Хвиля 2 — KPI по відділах:** `kpi_defs` (метрика × ціль × період × відділ),
  авто-підрахунок з БД сайту через count_news (новини/власні по авторах) —
  журналістка НЕ звітує, лічильник живе сам; прогрес-бари в апці; блок KPI
  у понеділковому «Тижневику Лиса»; закриті періоди — у `kpi_snapshots`.
- **Хвиля 3 — радість:** екран «Мої матеріали» (найкращі за переглядами з
  daily_stats.top_pages + article_stats, автор — get_author_from_url/owner_id);
  стріки, бейджі за серії виконаних KPI, конфеті вже є.
- **Хвиля 4 — зручності:** нагадування про дедлайни (scheduler), коментарі в
  картці, закриття таска лінком з авто-верифікацією по nodes, фільтри/пошук.

## Обмеження хвилі 1

- Пінг не дійде, доки людина не відкрила апку хоч раз (див. Ідентифікація).
- Дедлайн-нагадувань ще немає — тільки візуальне «прострочено» в апці.
- Статика віддається без кешу (no-cache на index) — швидкі ітерації важливіші
  за мілісекунди, обсяг апки < 50 КБ.
