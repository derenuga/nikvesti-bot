# Ревʼю панелі доступності demo.nikvesti.com

Дата: 21.07.2026. Оцінка реалізації фічі «Website adaptation for visually impaired users»
(грант IMS: WCAG 2.1 compliance, screen reader compatibility, high-contrast mode, testing with target users).

Аналізувався живий код demo.nikvesti.com: розмітка панелі (`#accessibility-panel`),
скрипт `AccessibilityManager` (інлайн у сторінці), стилі `css/style.css` (блок `am-*`).

**Супутній файл:** [`accessibility_panel_mockup.html`](accessibility_panel_mockup.html) —
робочий макет виправленої панелі (розмітка + ARIA + JS + CSS), з якого розробник сайту
може забирати код напряму.

---

## Загальний вердикт

Підхід правильний: **власна легка реалізація, а не сторонній оверлей** (accessiBe,
UserWay тощо). Це важливо — сторонні «кнопки доступності» мають погану репутацію в
a11y-спільноті (див. overlayfactsheet.com, підписаний сотнями експертів), і донору
власна реалізація виглядає сильніше.

Але в поточному вигляді є дві системні проблеми:

1. **Сама панель доступності не відповідає WCAG** — незрячий користувач зі скрінрідером
   не може нею скористатися (кнопки без імен, стан вибору не озвучується, фокус
   не утримується в модалці). Це головний ризик перед донором.
2. **Панель ≠ WCAG 2.1 compliance.** Обіцянка гранту закривається насамперед базовою
   розміткою сайту (семантика, alt-тексти, клавіатурна навігація, контраст дефолтної
   палітри, skip-link), а панель — додатковий інструмент поверх. Зараз на головній
   немає skip-link, і базовий аудит не проводився.

## A. Критичне: доступність самої панелі

| # | Проблема | Де | Виправлення |
|---|---|---|---|
| A1 | **Панель російською на сайті з `lang="uk"`.** Скрінрідер з українським голосом озвучує російський текст спотворено; на скрінах для донора це теж мінус | Усі рядки панелі («Настройки доступности», «Сбросить»…) | Словник рядків за мовою версії сайту (ua/ru/en). Дефолт — українська |
| A2 | **Кнопка-тогл без accessible name.** `.js-am-toggle` не має `aria-label`, SVG всередині `aria-hidden` → скрінрідер оголошує просто «кнопка». Людина, для якої фіча зроблена, не може її знайти | `<button class="icon-button js-am-toggle">` | `aria-label="Налаштування доступності"` + `aria-haspopup="dialog"` + `aria-controls="accessibility-panel"` |
| A3 | **Опції-іконки без підписів.** Кнопки міжрядкового інтервалу, інтервалу між літерами і шрифту — без `aria-label`; у кнопки «синьої» теми `aria-label=""` (порожній) | Групи `lineHeight`, `letterSpacing`, `fontFamily`, `theme` | Видимий текстовий підпис на кожній кнопці (див. A6), aria-label як мінімум |
| A4 | **Вибраний стан — лише CSS-клас `is-active`.** Скрінрідер не знає, яка опція увімкнена | `renderControls()` | Групи → `role="radiogroup"` + кнопки `role="radio"` + `aria-checked` + стрілочна навігація (roving tabindex). Мінімум — `aria-pressed` |
| A5 | **Немає focus trap.** `aria-modal="true"` заявлено, але Tab виходить із панелі на сторінку за бекдропом. Фокус-стилів (`:focus-visible`) у CSS панелі немає взагалі | `open()`/`close()`, CSS `am-*` | Утримувати Tab у межах панелі (цикл перший↔останній), видимий фокус-обвід на всіх контролах |
| A6 | **Іконки незрозумілі й зрячим.** «T із засічками / без» не читається; чорний кружечок 4-ї теми не каже нічого. Це і є головна причина «не подобається інтерфейс» | Усі групи опцій | Іконка + видимий текстовий підпис. Для шрифту — прев'ю самим шрифтом («Аа»), для тем — назва («Темна», «Контрастна») |

## B. Серйозне: як воно працює

| # | Проблема | Де | Виправлення |
|---|---|---|---|
| B1 | **Масштаб шрифту інлайн-стилями по кожному елементу** (`captureBaseStyles` + `style.fontSize=…!important`). Ламається на динамічному контенті: довантаження стрічки, пошук, таби — нові елементи лишаться дрібними до наступної зміни налаштувань | `applyTextOverrides()` | Масштабувати корінь: `html { font-size: calc(100% * scale) }` (сайт на rem), або CSS-змінна + селектори. Без JS-обходу DOM |
| B2 | **«Сховати зображення» ховає всі SVG**, крім `.am-keep-svg`, якого ніде не проставлено → зникають іконки інтерфейсу разом із самою кнопкою доступності. Плюс різнобій класів-винятків: `.am-skip-svg` (grayscale) vs `.am-keep-svg` (hide) | `body.am-images-hidden svg:not(.am-keep-svg)` | Ховати лише контентні зображення (`img`, `picture`, `figure`), SVG-іконки інтерфейсу не чіпати. Один клас-виняток |
| B3 | **Теми — переліком селекторів** (`.post-item-title, .tabs label, …`). Завжди буде «латка»: на скріні темної теми видно сірі непофарбовані зони. Кожен новий блок верстки треба буде дописувати в перелік | Блок `am-theme-*` у style.css (~300 рядків) | Перевести палітру сайту на CSS-змінні (`--bg`, `--text`, `--link`…) і в темі міняти лише значення змінних. Разова робота, знімає проблему назавжди |
| B4 | **Спалах білої теми при переходах.** Стан застосовується на `DOMContentLoaded` → користувач темної теми на кожній сторінці бачить білий спалах | ініціалізація | Інлайн-скрипт у `<head>`: прочитати localStorage і повісити клас теми на `<html>` до рендера |
| B5 | **«Лінія читання» працює тільки від миші** | `handleMouseMove` | Мінімум — рухати також за фокусом (`focusin`); або чесно позначити як інструмент для миші |
| B6 | Дрібне: у тогла SVG з `width="800px"` в атрибутах (рятує лише CSS); `viewBox="-1 0 14 14"` обрізаний; закоментований старий код у продакшн-розмітці | розмітка | Почистити |

## C. Чого не вистачає (дешеві доповнення)

- **Підкреслювати посилання** — класика для слабозорих, одна CSS-правка.
- **Зупинити анімації/рух** — це ще й вимога WCAG 2.2.2; заодно поважати `prefers-reduced-motion`.
- **Поважати системні налаштування** як дефолт: `prefers-color-scheme` (темна тема), `prefers-contrast`.
- **Ярлик тем зрозумілою мовою**: «Звичайна / Темна / Світла / Висока контрастність» замість безіменних кружечків.
- Кнопка тогла повинна бути **досяжна з клавіатури рано в Tab-порядку** (зараз вона в кінці DOM).

## D. Рівень сайту — те, що насправді і є «WCAG 2.1 compliance»

Панель не замінює базову доступність. Мінімальний чекліст по сайту:

1. **Skip-link** «Перейти до вмісту» — зараз відсутній.
2. **Alt-тексти** зображень (на головній 33 порожні `alt=""` — для декоративних це ок,
   для контентних ілюстрацій — ні; потрібен прохід редакційним оком + правило в CMS).
3. **Ієрархія заголовків** h1→h2→h3 без пропусків на шаблонах.
4. **Клавіатурна навігація**: меню, таби стрічки, каруселі, пошук — усе має працювати
   без миші, з видимим фокусом (глобальний `:focus-visible`).
5. **Контраст дефолтної палітри** — перевірити пари текст/фон на AA (4.5:1).
6. **Форми** (пошук, коментарі, донат): `label`, повідомлення про помилки текстом.
7. Автоматичний аудит **axe DevTools / Lighthouse Accessibility** по головних шаблонах
   (головна, стаття, рубрика, пошук) + ручний прохід **NVDA** (безкоштовний скрінрідер).

## E. Для звіту донору

- Пункт «testing with target users» — окрема обіцянка бюджету. Провести хоча б одну
  сесію з реальними користувачами (миколаївська організація УТОС — природний партнер),
  зафіксувати протокол і виправлення. Це найсильніший доказ у наративному звіті.
- Додати на сайт сторінку **«Заява про доступність»** (Accessibility statement):
  цільовий рівень WCAG 2.1 AA, перелік можливостей, контакт для скарг. Стандартна
  практика, донори це люблять, робиться за годину.
- У звіт: власна реалізація без сторонніх оверлеїв + результати axe/NVDA-аудиту
  «до/після» + протокол тестування з користувачами.

## Порядок робіт (пропозиція)

1. **Тиждень 1 — панель**: локалізація рядків, ARIA/фокус (A1–A6), підписи на кнопках
   за макетом `accessibility_panel_mockup.html`.
2. **Тиждень 1–2 — механіка**: масштаб через корінь (B1), фікс ховання SVG (B2),
   анти-спалах (B4), нові перемикачі (C).
3. **Тиждень 2–3 — теми через CSS-змінні** (B3) — найбільший шматок, але знімає
   «латки» назавжди.
4. **Тиждень 3–4 — базовий WCAG**: skip-link, фокуси, alt-и, axe/NVDA-аудит (D).
5. **Фінал**: сесія з користувачами + заява про доступність + матеріали для звіту (E).
